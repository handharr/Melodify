90-Minute Script

Interview Prep

Whiteboard design — what to draw and what to say, minute by minute.

Format

  • 90 minutes, whiteboard design of a mobile feature
  • Tool: Excalidraw — pick it and practice it tonight
  • 4 competencies: Clean Architecture · Design Patterns · Reasoning & Trade-offs · Monitoring & Observability
Opening line (always the same): "Before I draw anything, let me clarify requirements and constraints — then I'll define the API contract before touching any architecture boxes."

What They're Actually Evaluating

This interview is graded on a gradient to determine your level — not pass/fail. Three dimensions:

DimensionWhat they're watching for
Technical leadershipCan you break a fuzzy problem into coherent, scoped parts?
Architectural insightDo you understand the trade-offs behind each decision?
Product senseDo you think about the user, not just the system?
Mindset: Tech Lead, not Developer. You are a Tech Lead talking to a Product Manager. A Tech Lead doesn't start coding — they ask clarifying questions, categorize the work, and scope the problem before touching a whiteboard. A strong candidate doesn't find "the correct answer." They demonstrate how they think.

Trade-offs are not a section — they are your voice throughout. State a trade-off at the moment you make the decision. A senior candidate doesn't wait to be asked — they name the alternative and explain the choice as each box hits the board.

90-Minute Breakdown

0–10

Clarify First — Three Buckets

Ask across three explicit buckets and write their answers on the board.

Functional Requirements — what the app must do

  • Read-only or create/edit? Search or filtering?
  • How is the feed sorted — chronological or popularity?
  • Pull-to-refresh required? Can users act (like, comment) while offline?
  • Push notifications needed? Real-time — WebSocket, polling, or push?

Non-Functional Requirements — how the app must perform

  • Scale: DAUs? Which regions?
  • Performance: Target load speed, network budget, battery impact
  • Consistency: Eventual (likes, feed) or strong (payments, inventory)?
  • Offline: Stale data OK, or connectivity required? Read-only or sync-on-reconnect?
  • Media: Images, audio, video? Pagination or bounded list?

Out of Scope

Explicitly name what you won't cover. This signals seniority and protects your time.

"I'm asking these because each answer changes which patterns I choose."
10–25

API Design & Data Model

Define the contract before drawing any architecture boxes. Say:

"I'll sketch the endpoints and domain models first — that pins down what the layers need to move around."

Endpoints

GET /feed?cursor=&limit=20 → { items: [...], nextCursor } GET /tracks/{id} → Track POST /playlists → Playlist PUT /playlists/{id}/tracks → Playlist

Domain Models — what the app cares about (not DTOs yet)

Track { id, title, artist, durationMs, artworkURL } Playlist { id, name, tracks: [Track], ownerID } FeedItem (enum) { .banner(Banner) | .track(Track) | .playlist(Playlist) }

Trade-off — API style

OptionPick when
RESTSimple CRUD, well-defined resources — the default choice
GraphQLFlexible, nested, or multi-resource queries; client-driven shape
gRPCHigh-throughput, contract-first internal services

Trade-off — pagination

OptionPick when
Cursor-basedInfinite-scroll feeds — rows shift as new content is inserted between pages
Offset-basedPaginated lists where total count matters and order is stable

Trade-off — real-time updates

Polling is simpler infra and fine for low-frequency updates (social counts, feed refresh). WebSocket for live playback state, collaborative editing, or chat. State the choice and the reason.

25–40

High-Level Architecture — Draw the Layers

Draw the four-layer stack. As you place each layer, state the dependency rule and the decision that shaped it.

"Domain depends on nothing. That's the rule everything else flows from."
Presentation → ViewController + ViewModel (@Published) Domain → UseCase + Protocol + Model + Param + FetchPolicy Data → Repository + DataSource + DTO + Mapper + APIClient Application → AppDelegate + Coordinator + DI wiring

Trade-offs — state these as you draw each layer

DecisionSay this
UIKit vs SwiftUISwiftUI is the modern default — faster for declarative state-driven UI. UIKit gives more control over cell reuse, custom layouts, and transitions. I'd ask which the team uses before assuming.
MVVM vs MVP vs MVIMVVM with @Published is reactive and familiar to iOS teams. MVP makes the Presenter easier to unit test without a framework. MVI gives strict unidirectional flow for complex state. Best choice is what the team uses consistently.
Combine vs async/await@Published + sink for ViewModel state. async/await for one-shot sequential async work. I don't mix them for the same operation. Combine shines with debounce/removeDuplicates pipelines.
CoordinatorRemoves navigation ownership from ViewControllers — reusable and independently testable. Trade-off: added indirection. For a simple app it can be unnecessary ceremony.
UIModelCarries display-ready data — formatted strings, localized text. If the Domain model reaches ViewController, formatting logic leaks into the View layer.
40–60

Trace the Full Data Flow

Pick the hardest path and walk it end-to-end out loud. Name trade-offs at the moment you reach each decision — not in a separate section later.

"User opens the app. AppCoordinator builds the dependency graph — one APIClient, repositories injected into UseCases, UseCases injected into ViewModels. ViewModel calls FetchHomeDataUseCase with FetchPolicy.cached. UseCase fires async let for tracks and playlists concurrently. Each repo checks policy: cache-first, fall back to network. Response comes back as a DTO, Mapper converts it to a Domain model, ViewModel maps it to a UIModel. UIModel is the only type that reaches the ViewController. ViewController sinks on @Published feedItems — a HomeFeedItem enum — and switches on each case. Compiler enforces exhaustiveness, no casting."

Trade-off — caching stack (name this as you trace the data path)

HTTP cache-control + ETag / Last-Modified ← server collaboration, free freshness ↓ Memory cache (NSCache) ← fast, auto-evicted under memory pressure ↓ Disk cache ← persists across sessions ↓ Custom invalidation strategy ← TTL, version key, manual purge

Trade-off — storage (at the Data layer)

StorageWhen to use
UserDefaultsSimple key-value, small payloads, preferences, read-through cache of flat API responses
Core DataRelational data, relationships, migrations, complex queries
SQLite / GRDBLightweight relational without Core Data overhead
KeychainCredentials, tokens — anything requiring secure storage
File system / disk cacheLarge blobs — images, audio, downloaded tracks
60–75

Senior-Level Deep Dives (Unprompted)

Mobile-specific constraints — call these out before being asked:

  • Battery: minimize background work and radio usage; batch network calls, avoid polling when push is viable
  • Memory: handle large datasets without bloating RAM — cell reuse, image caching with eviction policies
  • Network: be deliberate about what you fetch and when — delta sync, pagination, HTTP cache headers

Observability

"AnalyticsServiceProtocol lives in Domain so ViewModels can call it without a layer violation. Events are typed enums with associated values — compiler catches typos. Swap ConsoleAnalyticsService for Firebase or Amplitude without touching business logic."

Error handling

"Repository errors are typed — PlaylistError.notFound, APIError.noNetwork. ViewModel maps them to user-facing strings at the UIModel boundary. ViewController never sees a raw error."

Deep links + push notifications

"Both route through a single DeepLinkHandler on AppCoordinator. URL opens and push taps converge at one switch — one routing table, not two. DeepLinkParser is a pure function, fully testable with no mocks."

Offline sync — if in scope, name these concerns:

  • Conflict resolution: last-write-wins, server-wins, or merge strategy?
  • Request queuing: persist write ops locally until connectivity returns
  • Background retries: exponential backoff, idempotency keys on mutations
  • Delta sync: only fetch what changed since last sync (timestamp / cursor)
  • Batched requests: coalesce queued ops into one call
  • Resumable uploads/downloads: chunked transfer with local byte offset

Testing

"Every layer is tested against the layer below via mock protocols. I never mock the concrete type — I mock the protocol. That's what makes the architecture testable, not just tested."
75–90

Close Strong

"To recap — Domain is pure business logic with no dependencies. Data implements Domain protocols, network and cache are behind their own interfaces. Presentation only holds UIModels. Application wires the dependency graph at startup. Every boundary is a protocol — independently testable and swappable."

Then ask them (pick 1–2):

  • Is real-time sync a future requirement? That changes how I'd structure cache invalidation.
  • What does the existing analytics stack look like? I want to know what I'd be adapting to.
  • What's the biggest pain point in the current architecture that this redesign should address?

Time Management Warning

API design is the most likely bottleneck. Mobile engineers are consumers of APIs at work, not designers. This makes the API design phase the one most likely to run long. Practice it independently with a timer. When you notice you're going deep on one section, say explicitly: "I want to make sure we get to the high-level design — let me summarize and move on."

Knowledge Topology — Study Checklist

Know where your gaps are. Each cluster is a topic to master independently.

Mobile Domain

UIKit vs SwiftUI
SwiftUI + Declarative, less boilerplate, live Xcode previews, auto dark mode & accessibility − Still maturing; gaps in complex cell reuse, custom transitions, fine-grained scroll control UIKit + Battle-tested, full control over layout, transitions, and cell lifecycle − Verbose; navigation logic bleeds into ViewController at scale Recommended: SwiftUI for new greenfield screens. UIKit when joining an existing team or needing complex list performance. Always ask what the team uses first.
App lifecycle
foreground — active, full CPU. background — limited CPU, no UI updates, ~30s to finish work. suspended — frozen in memory, killed without notice.

+ Know what's safe in each state: network calls OK in background, UI mutations are not − Background tasks not properly ended → app gets terminated by watchdog Recommended: Use beginBackgroundTask to request extra time for critical work. Always call endBackgroundTask when done — even on failure.
GCD vs OperationQueue vs async/await
async/await + Readable, structured concurrency, compiler-enforced — default for new code − Can't express complex dependency graphs between tasks GCD + Simple, low overhead for fire-and-forget background work − No cancellation, callback nesting gets messy fast OperationQueue + Built-in cancellation, task dependencies, max concurrency limit − More boilerplate than the alternatives Recommended: async/await for nearly everything new. GCD only for one-off background dispatch. OperationQueue when you need explicit cancellation or inter-task dependencies (e.g. image pipeline).
Navigation: stack, tab, deep link
Stack — drill-down hierarchy. Tab — peer sections at the same level.

+ Coordinator centralises routing: one routing table, independently testable, no nav logic in VCs − Added indirection; over-engineering for a simple two-screen app Recommended: Coordinator for any app with deep linking, auth flows, or 3+ screens. Skip it for a simple demo with a single navigation path.

API Design & Networking

REST vs WebSockets vs GraphQL vs gRPC
REST + Simple, universally understood, excellent tooling and caching support − Over/under-fetching; multiple round trips for related resources WebSockets + Persistent, low-latency bidirectional channel — ideal for real-time features − Stateful connection is harder to scale and debug; needs reconnect logic GraphQL + Client-defined shape, single endpoint, no over-fetching − Complex caching, N+1 query risk server-side, steeper learning curve gRPC + Strongly typed contracts, efficient binary encoding, built-in streaming − Not human-readable, overkill for most mobile-to-backend calls Recommended: REST as the default. WebSockets when you need live push from server to client. GraphQL when the client drives query shape (BFF pattern). gRPC for internal high-throughput microservices.
Cursor vs offset pagination
Cursor-based + Stable with live data — no skipped or duplicate rows when new items are inserted mid-scroll − Can't jump to an arbitrary page; no total count Offset-based + Can jump to any page, easy to show total count and page numbers − Rows shift between pages when data is inserted — causes skips and duplicates in feeds Recommended: Cursor for any live feed (home, activity, search results). Offset for admin panels, static lists, or anywhere "go to page 5" is a real user need.
Full caching stack
HTTP cache-control + ETag — server signals freshness, client reuses response for free
NSCache (memory) — sub-millisecond reads, auto-evicted under memory pressure
Disk cache — persists across app restarts, slower reads
Custom invalidation — TTL, version key, or manual purge for cases the above can't handle

+ Each layer handles a different freshness/speed trade-off — name all four to signal depth − Stale data risk grows as you move down the stack; invalidation strategy must be explicit Recommended: Start with HTTP headers (free). Add NSCache for images and repeated reads. Add disk only when offline support is required. Add custom invalidation only when TTL or versioning is needed.
Offline sync conflict resolution
Last-write-wins + Simple, no coordination needed — fine for likes, play counts, preferences − Silent data loss if two clients write concurrently Server-wins + Authoritative and safe — correct for payments, inventory, financial data − User's local changes are silently discarded, which can feel broken Merge strategy + Fair — both sides' intent is preserved (e.g. collaborative doc editing) − Complex to implement, hard to reason about edge cases Recommended: Last-write-wins for social/preference data. Server-wins for anything financial or authoritative. Merge only if the feature explicitly requires preserving concurrent edits (notes, docs, whiteboards).
Authentication: token refresh + storage
Short-lived access token + long-lived refresh token. Refresh silently on 401.

Keychain + Encrypted, sandboxed, survives app reinstall — required for tokens and passwords − Slower API, verbose boilerplate UserDefaults + Simple API, fine for non-sensitive preferences − Unencrypted plist on disk — never store tokens here Recommended: Always Keychain for tokens. Short-lived access token + long-lived refresh token. Refresh silently on 401 — the user should never see an auth error mid-session.
Retry policy + exponential backoff
Retry on transient failures (5xx, timeout) with increasing delay: 1s → 2s → 4s → give up.

+ Prevents thundering herd on the server; resilient to brief outages − Delayed feedback to the user; mutations must carry idempotency keys to prevent duplicate writes on retry Recommended: Always apply exponential backoff on network failures. Cap at 3–5 retries. Show a user-facing error after the last attempt — don't retry silently forever.

Architecture

Clean Architecture layer direction
Presentation → Domain ← Data. The arrow always points inward.

+ Domain has zero framework imports — fully testable with plain Swift, portable across platforms − More files and interfaces than a flat structure; feels like overhead on small features Recommended: Always enforce the dependency rule, even in small apps. The habit is cheap to build early and expensive to retrofit later.
MVVM vs MVC vs VIPER
MVVM + Testable ViewModel, reactive state with Combine/async — iOS team default − ViewModel can grow large without discipline (add UseCases to keep it thin) MVC + Minimal boilerplate, fast to write, Apple's native pattern − Massive ViewController syndrome; navigation and business logic bleed in VIPER + Strict role separation — every concern has one file, easy to enforce in large teams − Very verbose; ceremony cost is too high for most features Recommended: MVVM + Clean Architecture for most iOS work. MVC only for throwaway prototypes. VIPER only if the existing codebase already uses it or the team is very large with strict ownership boundaries.
Constructor injection vs DI container
Constructor injection + Explicit dependencies — easy to trace, test, and reason about − Wiring boilerplate grows in large modular apps DI container (e.g. Swinject, Factory) + Less boilerplate at scale, centralised registration − Dependencies are implicit — harder to see what a type actually needs at a glance Recommended: Constructor injection always. Only reach for a DI container when the wiring boilerplate across many modules becomes genuinely painful.
Repository pattern
UseCase calls repository.fetch() — it doesn't know if that's network, cache, or disk.

+ Swap data source (e.g. mock for tests) without touching business logic; single place to own caching strategy − Extra abstraction layer; for a truly trivial data source it can feel like unnecessary indirection Recommended: Always use Repository in Clean Architecture. It's the pattern that makes FetchPolicy and cache/network fallback clean to implement.

Storage

UserDefaults vs Core Data vs SQLite vs Keychain vs disk
UserDefaults + Trivial API, fine for preferences and small flat payloads − Unencrypted plist; no relational queries; not for secrets or large data Core Data + Powerful: relationships, migrations, NSFetchedResultsController, faulting − Complex setup, steep learning curve, overkill for simple caching SQLite / GRDB + Relational queries without Core Data ceremony; lightweight and fast − Raw SQL or DSL required; no Apple-native tooling Keychain + Encrypted, sandboxed, survives reinstall — the only correct choice for tokens − Verbose C-style API (wrap it); slower than UserDefaults Disk / file cache + Right for large blobs: images, audio, downloaded tracks − Manual eviction; must manage file paths and cache size yourself Recommended: Match storage to data type — UserDefaults for settings, Core Data for relational, Keychain for secrets, disk for blobs. Never use UserDefaults as a general-purpose database.
When Keychain is required vs acceptable risk
Keychain required — auth tokens, passwords, biometric keys, any PII UserDefaults OK — UI state, feature flags, preferences, non-sensitive read-through cache

+ Keychain items are encrypted at rest and locked to the app's sandbox − Using UserDefaults for tokens is a common security finding in audits — don't do it Recommended: Default to Keychain for anything that grants access or identifies the user. If in doubt, ask: "would I be comfortable if this value appeared in a crash log?" If no — use Keychain.

Performance

Memory: retain cycles
Closure captures self strongly → cycle if the object also holds that closure (e.g. stored Combine sink).

[weak self] + Safe — becomes nil when the object is deallocated; no crash − Requires guard let self boilerplate inside the closure [unowned self] + No optional unwrap needed − Crashes if self is already deallocated — only safe when lifetimes are guaranteed equal Recommended: Default to [weak self] everywhere. Use [unowned self] only when you can prove the closure cannot outlive the object (rare). Detect leaks with Instruments → Leaks or Xcode Memory Graph.
Battery drain
Top culprits: polling, continuous GPS, background fetch, frequent small network requests keeping the radio awake.

Push over polling + Radio stays idle until the server has something to say — massive battery win − Requires server-side push infrastructure; polling is simpler for MVP Batch requests + One radio wake-up for N calls instead of N wake-ups − Adds latency if some calls are time-sensitive Recommended: Push notifications over polling whenever the backend supports it. Batch non-urgent analytics and logging calls. Use significant-location-change instead of continuous GPS unless precision is critical.
60 FPS / main thread
Each frame has 16ms budget. Layout, drawing, and @MainActor state updates all run here.

+ Moving image decoding, JSON parsing, and disk I/O to a background queue keeps the main thread free − Forgetting to dispatch back to main for UI updates causes purple thread-checker warnings (and crashes) Recommended: Mark ViewModels @MainActor — all state mutations automatically happen on main. Use async functions for background work and await back on the actor. Profile with Instruments → Time Profiler.
Cold start vs warm start
Cold start — process launched from scratch; dylibs loaded, AppDelegate called. Most expensive. Warm start — app already in memory, brought to foreground. Very fast.

+ Deferring analytics init, non-critical network calls, and heavy data loading until after first frame renders noticeably improves perceived startup − Too much deferred work can cause a visible content "pop-in" after launch Recommended: In application(_:didFinishLaunching) do only what's required for the first screen. Defer analytics, logging SDKs, and prefetch calls until after viewDidAppear.

Observability & Testing

Unit test structure: mock protocol, not concrete type
UseCase test mocks RepositoryProtocol. ViewModel test mocks UseCaseProtocol.

+ Implementation can change freely without breaking tests; each layer is tested in isolation − Must maintain mock implementations alongside real ones (code-gen tools like Mockolo help) Recommended: One mock per protocol, kept next to the test target. Never use XCTestExpectation with a hardcoded timeout — use async/await tests instead for deterministic results.
Sentry vs Firebase Crashlytics
Firebase Crashlytics + Free, tight Firebase integration, zero-config setup, great symbolication − Limited to crash reporting; performance monitoring is a separate Firebase product Sentry + Captures non-fatal errors, breadcrumbs, performance traces, and crashes in one SDK; self-hostable − Paid beyond the free tier; slightly more integration work Recommended: Crashlytics if you're already in the Firebase ecosystem. Sentry if you want non-crash error tracking and performance monitoring without adding multiple SDKs.

Privacy & Security

Encryption at rest vs in transit
In transit — TLS (HTTPS) + Non-negotiable; App Transport Security enforces it by default on iOS − ATS exceptions exist — make sure none were added without a reason At rest — iOS Data Protection + Files with .completeFileProtection are encrypted when the device is locked; Keychain adds another layer for secrets − Default file protection is .completeUntilFirstUserAuthentication — weaker; must opt up explicitly for sensitive files Recommended: Enable .completeFileProtection for any file containing user data. Always use HTTPS — no exceptions. Store secrets in Keychain, not in files.
iOS permissions model
+ Ask at the moment of use (contextual) — grant rates are significantly higher than asking on launch − iOS gives only one native prompt per permission; if denied, the user must go to Settings manually Recommended: Show a pre-prompt explaining why you need the permission before triggering the system dialog — you only get one shot. Always degrade gracefully if denied; never block the core flow.
Certificate pinning
Rejects any TLS cert not matching the pinned one — even a cert from a trusted CA is blocked.

+ Prevents MITM attacks even if a CA is compromised; strong guarantee for sensitive data in transit − If the cert rotates and the pinned build is still in the wild, those users get network failures until they update. Requires a cert rotation plan and a grace-period fallback. Recommended: Use certificate pinning only for high-security apps (banking, healthcare). For most apps, HTTPS + ATS is sufficient. If you pin, pin the public key (not the full cert) and support multiple pins simultaneously to allow rotation.
