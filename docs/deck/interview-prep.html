90-Minute Script

Interview Prep

Whiteboard design — what to draw and what to say, minute by minute.

Format

  • 90 minutes, whiteboard design of a mobile feature
  • Tool: Excalidraw — pick it and practice it tonight
  • 4 competencies: Clean Architecture · Design Patterns · Reasoning & Trade-offs · Monitoring & Observability
Opening line (always the same): "Before I draw anything, let me clarify requirements and constraints — then I'll define the API contract before touching any architecture boxes."

What They're Actually Evaluating

This interview is graded on a gradient to determine your level — not pass/fail. Three dimensions:

DimensionWhat they're watching for
Technical leadershipCan you break a fuzzy problem into coherent, scoped parts?
Architectural insightDo you understand the trade-offs behind each decision?
Product senseDo you think about the user, not just the system?
Mindset: Tech Lead, not Developer. You are a Tech Lead talking to a Product Manager. A Tech Lead doesn't start coding — they ask clarifying questions, categorize the work, and scope the problem before touching a whiteboard. A strong candidate doesn't find "the correct answer." They demonstrate how they think.

Trade-offs are not a section — they are your voice throughout. State a trade-off at the moment you make the decision. A senior candidate doesn't wait to be asked — they name the alternative and explain the choice as each box hits the board.

90-Minute Breakdown

0–10

Clarify First — Three Buckets

Ask across three explicit buckets and write their answers on the board.

Functional Requirements — what the app must do

  • Read-only or create/edit? Search or filtering?
  • How is the feed sorted — chronological or popularity?
  • Pull-to-refresh required? Can users act (like, comment) while offline?
  • Push notifications needed? Real-time — WebSocket, polling, or push?

Non-Functional Requirements — how the app must perform

  • Scale: DAUs? Which regions?
  • Performance: Target load speed, network budget, battery impact
  • Consistency: Eventual (likes, feed) or strong (payments, inventory)?
  • Offline: Stale data OK, or connectivity required? Read-only or sync-on-reconnect?
  • Media: Images, audio, video? Pagination or bounded list?

Out of Scope

Explicitly name what you won't cover. This signals seniority and protects your time.

"I'm asking these because each answer changes which patterns I choose."
10–25

API Design & Data Model

Define the contract before drawing any architecture boxes. Say:

"I'll sketch the endpoints and domain models first — that pins down what the layers need to move around."

Endpoints

GET /feed?cursor=&limit=20 → { items: [...], nextCursor } GET /tracks/{id} → Track POST /playlists → Playlist PUT /playlists/{id}/tracks → Playlist

Domain Models — what the app cares about (not DTOs yet)

Track { id, title, artist, durationMs, artworkURL } Playlist { id, name, tracks: [Track], ownerID } FeedItem (enum) { .banner(Banner) | .track(Track) | .playlist(Playlist) }

Trade-off — API style

OptionPick when
RESTSimple CRUD, well-defined resources — the default choice
GraphQLFlexible, nested, or multi-resource queries; client-driven shape
gRPCHigh-throughput, contract-first internal services

Trade-off — pagination

OptionPick when
Cursor-basedInfinite-scroll feeds — rows shift as new content is inserted between pages
Offset-basedPaginated lists where total count matters and order is stable

Trade-off — real-time updates

Polling is simpler infra and fine for low-frequency updates (social counts, feed refresh). WebSocket for live playback state, collaborative editing, or chat. State the choice and the reason.

25–40

High-Level Architecture — Draw the Layers

Draw the four-layer stack. As you place each layer, state the dependency rule and the decision that shaped it.

"Domain depends on nothing. That's the rule everything else flows from."
Presentation → ViewController + ViewModel (@Published) Domain → UseCase + Protocol + Model + Param + FetchPolicy Data → Repository + DataSource + DTO + Mapper + APIClient Application → AppDelegate + Coordinator + DI wiring

Trade-offs — state these as you draw each layer

DecisionSay this
UIKit vs SwiftUISwiftUI is the modern default — faster for declarative state-driven UI. UIKit gives more control over cell reuse, custom layouts, and transitions. I'd ask which the team uses before assuming.
MVVM vs MVP vs MVIMVVM with @Published is reactive and familiar to iOS teams. MVP makes the Presenter easier to unit test without a framework. MVI gives strict unidirectional flow for complex state. Best choice is what the team uses consistently.
Combine vs async/await@Published + sink for ViewModel state. async/await for one-shot sequential async work. I don't mix them for the same operation. Combine shines with debounce/removeDuplicates pipelines.
CoordinatorRemoves navigation ownership from ViewControllers — reusable and independently testable. Trade-off: added indirection. For a simple app it can be unnecessary ceremony.
UIModelCarries display-ready data — formatted strings, localized text. If the Domain model reaches ViewController, formatting logic leaks into the View layer.
40–60

Trace the Full Data Flow

Pick the hardest path and walk it end-to-end out loud. Name trade-offs at the moment you reach each decision — not in a separate section later.

"User opens the app. AppCoordinator builds the dependency graph — one APIClient, repositories injected into UseCases, UseCases injected into ViewModels. ViewModel calls FetchHomeDataUseCase with FetchPolicy.cached. UseCase fires async let for tracks and playlists concurrently. Each repo checks policy: cache-first, fall back to network. Response comes back as a DTO, Mapper converts it to a Domain model, ViewModel maps it to a UIModel. UIModel is the only type that reaches the ViewController. ViewController sinks on @Published feedItems — a HomeFeedItem enum — and switches on each case. Compiler enforces exhaustiveness, no casting."

Trade-off — caching stack (name this as you trace the data path)

HTTP cache-control + ETag / Last-Modified ← server collaboration, free freshness ↓ Memory cache (NSCache) ← fast, auto-evicted under memory pressure ↓ Disk cache ← persists across sessions ↓ Custom invalidation strategy ← TTL, version key, manual purge

Trade-off — storage (at the Data layer)

StorageWhen to use
UserDefaultsSimple key-value, small payloads, preferences, read-through cache of flat API responses
Core DataRelational data, relationships, migrations, complex queries
SQLite / GRDBLightweight relational without Core Data overhead
KeychainCredentials, tokens — anything requiring secure storage
File system / disk cacheLarge blobs — images, audio, downloaded tracks
60–75

Senior-Level Deep Dives (Unprompted)

Mobile-specific constraints — call these out before being asked:

  • Battery: minimize background work and radio usage; batch network calls, avoid polling when push is viable
  • Memory: handle large datasets without bloating RAM — cell reuse, image caching with eviction policies
  • Network: be deliberate about what you fetch and when — delta sync, pagination, HTTP cache headers

Observability

"AnalyticsServiceProtocol lives in Domain so ViewModels can call it without a layer violation. Events are typed enums with associated values — compiler catches typos. Swap ConsoleAnalyticsService for Firebase or Amplitude without touching business logic."

Error handling

"Repository errors are typed — PlaylistError.notFound, APIError.noNetwork. ViewModel maps them to user-facing strings at the UIModel boundary. ViewController never sees a raw error."

Deep links + push notifications

"Both route through a single DeepLinkHandler on AppCoordinator. URL opens and push taps converge at one switch — one routing table, not two. DeepLinkParser is a pure function, fully testable with no mocks."

Offline sync — if in scope, name these concerns:

  • Conflict resolution: last-write-wins, server-wins, or merge strategy?
  • Request queuing: persist write ops locally until connectivity returns
  • Background retries: exponential backoff, idempotency keys on mutations
  • Delta sync: only fetch what changed since last sync (timestamp / cursor)
  • Batched requests: coalesce queued ops into one call
  • Resumable uploads/downloads: chunked transfer with local byte offset

Testing

"Every layer is tested against the layer below via mock protocols. I never mock the concrete type — I mock the protocol. That's what makes the architecture testable, not just tested."
75–90

Close Strong

"To recap — Domain is pure business logic with no dependencies. Data implements Domain protocols, network and cache are behind their own interfaces. Presentation only holds UIModels. Application wires the dependency graph at startup. Every boundary is a protocol — independently testable and swappable."

Then ask them (pick 1–2):

  • Is real-time sync a future requirement? That changes how I'd structure cache invalidation.
  • What does the existing analytics stack look like? I want to know what I'd be adapting to.
  • What's the biggest pain point in the current architecture that this redesign should address?

Time Management Warning

API design is the most likely bottleneck. Mobile engineers are consumers of APIs at work, not designers. This makes the API design phase the one most likely to run long. Practice it independently with a timer. When you notice you're going deep on one section, say explicitly: "I want to make sure we get to the high-level design — let me summarize and move on."

Knowledge Topology — Study Checklist

Know where your gaps are. Each cluster is a topic to master independently.

Mobile Domain

UIKit vs SwiftUI
SwiftUI — declarative, faster to build, great for state-driven screens.
UIKit — full control over cell reuse, transitions, complex layouts. Still the safe pick for teams on existing codebases.
Ask which the team uses before assuming.
App lifecycle
foreground → active user interaction. background → limited CPU, no UI. suspended → frozen, may be killed.
Know what code is safe to run in each state and what must be deferred or cancelled.
GCD vs OperationQueue vs async/await
async/await — readable sequential async, use for most new code.
GCD — low-level fire-and-forget dispatch, fine for simple background work.
OperationQueue — when you need cancellation, dependencies between tasks, or max concurrency.
Navigation: stack, tab, deep link
Stack → drill-down hierarchy. Tab → peer sections at the same level.
Deep link always routes through one handler (Coordinator) — not scattered across ViewControllers. One routing table beats two.

API Design & Networking

REST vs WebSockets vs GraphQL vs gRPC
REST — simple CRUD over well-defined resources. Default choice.
WebSockets — persistent connection for real-time (live playback state, chat, collaboration).
GraphQL — client-driven shape, flexible nested queries. Avoids over/under-fetching.
gRPC — typed contracts, high-throughput, internal service-to-service.
Cursor vs offset pagination
Cursor — for infinite-scroll feeds. Rows shift as new items are inserted, so offset skips or repeats items.
Offset — for stable lists where total count matters (search results, admin tables).
Full caching stack
HTTP cache-control + ETag → free freshness via server collaboration.
NSCache (memory) → fast, auto-evicted under memory pressure.
Disk cache → survives app restarts.
Custom invalidation → TTL, version key, or manual purge when the above aren't enough.
Offline sync conflict resolution
Last-write-wins — simple, good enough for most social data (likes, play counts).
Server-wins — authoritative, safe for inventory or financial data.
Merge strategy — complex but fair; needed for collaborative editing.
Authentication: token refresh + storage
Short-lived access token + long-lived refresh token. Refresh silently when access token expires.
Always store tokens in Keychain — never UserDefaults. UserDefaults is unencrypted and readable.
Retry policy + exponential backoff
Retry failed requests with increasing delay (1s → 2s → 4s…) to avoid thundering herd on the server.
Add idempotency key on mutations so a retried write doesn't create duplicates.

Architecture

Clean Architecture layer direction
Dependency arrow always points inward: Presentation → Domain ← Data.
Domain knows nothing about UIKit, URLSession, or any framework. That's what makes it testable and portable.
MVVM vs MVC vs VIPER
MVVM — testable ViewModel, reactive with Combine. iOS teams default.
MVC — simple, fast to write. Navigation logic bleeds into ViewController at scale.
VIPER — fully separated, every role has one file. Very verbose; justified only for large teams with strict boundaries.
Constructor injection vs DI container
Constructor injection — explicit, easy to trace, easier to test. Preferred for most cases.
DI container — less boilerplate at scale, but harder to see what depends on what. Worth it in large modular apps.
Repository pattern
Hides where data comes from (network vs cache vs disk) behind a single interface.
UseCase calls repository.fetch() — it doesn't know or care if that hits the network or a local cache.

Storage

UserDefaults vs Core Data vs SQLite vs Keychain vs disk
UserDefaults — preferences, feature flags, small flat payloads. Not for secrets.
Core Data — relational data with relationships, migrations, complex queries.
SQLite / GRDB — relational without Core Data ceremony. Good middle ground.
Keychain — tokens, passwords, anything sensitive. Survives app reinstall.
Disk / file cache — large blobs: images, audio files, downloaded tracks.
When Keychain is required vs acceptable risk
Keychain required — auth tokens, passwords, biometric keys, PII.
UserDefaults OK — preferences, UI state, feature flags, non-sensitive cache. Unencrypted, so anything readable by the user is fine here.

Performance

Memory: retain cycles
Closures capture self strongly by default → cycle if the object also holds the closure.
Fix: [weak self]. Detect: Instruments → Leaks / Memory Graph Debugger.
Battery drain
Main culprits: polling, GPS, background fetch, keeping the radio alive with frequent small requests.
Fix: push over polling, batch requests, use significant-location-change instead of continuous GPS.
60 FPS / main thread
Layout and drawing must complete in 16ms per frame. Anything slow on the main thread drops frames.
Move image decoding, JSON parsing, and disk I/O to a background queue. Profile with Instruments → Time Profiler.
Cold start vs warm start
Cold — process launched from scratch. Everything loads. Most expensive.
Warm — app already in memory, just brought to foreground. Fast.
Defer analytics init, non-critical network calls, and heavy data loading until after first frame.

Observability & Testing

Unit test structure
Mock the layer directly below the one you're testing — never the concrete type, always the protocol.
This means a UseCase test mocks the Repository protocol. A ViewModel test mocks the UseCase protocol.
Sentry vs Firebase Crashlytics
Crashlytics — tight Firebase integration, simpler setup, free.
Sentry — more control, self-hostable, richer error context, better for non-crash errors. Better choice if you care about performance monitoring too.

Privacy & Security

Encryption at rest vs in transit
In transit — TLS. All network calls must use HTTPS. Non-negotiable.
At rest — iOS Data Protection encrypts files when the device is locked. Use .completeFileProtection for sensitive data. Keychain adds an extra layer for secrets.
iOS permissions model
Ask at the moment of use — contextual requests have far higher grant rates than asking on launch.
Always degrade gracefully if denied: don't crash or block the core flow, just disable the feature that needs it.
Certificate pinning
Prevents MITM attacks by rejecting any cert not matching the pinned one — even if issued by a trusted CA.
Trade-off: if the cert rotates and you ship a pinned build, the app breaks until users update. Requires a rotation plan.

Mid-level answers the question asked.
