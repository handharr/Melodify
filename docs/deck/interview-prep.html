90-Minute Script

Interview Prep

Whiteboard design — what to draw and what to say, minute by minute.

Format

  • 90 minutes, whiteboard design of a mobile feature
  • Tool: Excalidraw — pick it and practice it tonight
  • Problem: Chat App / Music Streaming / Social Media Feed
Opening line (always the same): "Before I draw anything, let me clarify requirements and constraints — then I'll start with the hardest technical piece."

90-Minute Breakdown

0–10

Clarify Before Drawing

Ask and write their answers on the board.

  • Functionality: Read-only or create/edit? Search or filtering? Real-time — WebSocket, polling, or push?
  • Offline: Work offline with stale data, or connectivity required? Read-only offline or sync-on-reconnect?
  • Scale: Pagination, infinite scroll, or bounded? Media — images, audio, video?
  • Platform: iOS only? UIKit or SwiftUI preference?
"I'm asking these because each answer changes which patterns I choose."
10–25

Draw the Layers First

Draw the architecture before the feature. Then say:

"Domain depends on nothing. That's the rule everything else flows from."

Presentation → ViewController + ViewModel (@Published) Domain → UseCase + Protocol + Model + Param Data → Repository + DataSource + DTO + Mapper Application → AppDelegate + Coordinator + DI

25–50

Trace the Full Data Flow

Pick the hardest piece and walk it end-to-end out loud:

"User opens the app. The coordinator or app entry point builds the dependency graph. The ViewModel calls FetchHomeDataUseCase with FetchPolicy.cached. The UseCase fires async let for tracks and playlists concurrently. Each repo checks policy: cache-first, fall back to network. Response comes back as a DTO, Mapper converts it to a Domain model, ViewModel maps it to a UIModel. UIModel is the only type that reaches the View layer. The View binds to published state — a HomeFeedItem enum — and switches on each case. Compiler enforces exhaustiveness, no casting."
50–65

Add the Senior-Level Layers (Unprompted)

Observability

"I'd wire an AnalyticsServiceProtocol into the Domain layer. Events are typed enums with associated values — compiler catches typos. ConsoleAnalyticsService in Application layer — swap for Firebase or Segment without touching business logic."

Error handling

"Repository errors are typed. ViewModel maps them to user-facing strings at the UIModel boundary. ViewController never sees a raw error."

Deep links + push notifications

"Both route through a single DeepLinkHandler on AppCoordinator. URL opens and push taps converge at one switch — one routing table. DeepLinkParser is a pure function, fully testable."

Testing

"Every layer is tested against the layer below via mock protocols. I never mock the concrete type — I mock the protocol. That's what makes the architecture testable, not just tested."
65–80

Handle Trade-off Questions

They askYour answer
"UIKit or SwiftUI?"Both are valid — depends on team's stack and iOS version floor. SwiftUI is the modern default, faster for declarative state-driven UI. UIKit gives more control over custom layouts, cell reuse, and transitions. Worth asking before assuming.
"Combine or async/await?"Both are valid and compose well together. async/await excels at sequential async work with structured cancellation. Combine excels when multiple UI elements react to the same state, or when you need operators like debounce or removeDuplicates. Many teams use both — the key is not mixing them for the same operation without a clear reason.
"Why UIModel instead of Domain model in ViewController?"UIModel carries display-ready data — formatted strings, localized text. If Domain model reaches the ViewController, formatting logic leaks into business logic. That said, for simple screens a 1:1 mapping may not need a dedicated type — it's a trade-off between purity and ceremony.
"UserDefaults or Core Data?"Depends on the data shape and access patterns. UserDefaults + Codable is simple and sufficient for read-through caching of flat API responses. Core Data is the right fit when you need relational queries, complex migrations, or large datasets. Neither is universally better.
"MVVM, MVP, or MVI?"All three achieve separation of concerns — the meaningful difference is in how state flows. MVVM with @Published is reactive and familiar to iOS teams. MVP makes the Presenter easier to unit test without a framework. MVI gives strict unidirectional flow which helps in complex state scenarios. The best choice is often what the team already uses consistently.
"Why Coordinator?"Coordinators remove navigation ownership from ViewControllers, making them independently reusable and testable. The trade-off is added indirection — for simple apps it can be unnecessary ceremony. Alternatives like flow controllers or enum-driven routing solve the same problem differently. Worth naming the trade-off before recommending one.
80–90

Close Strong

"To recap — Domain is pure business logic with no dependencies. Data implements Domain protocols, network and cache are behind their own interfaces. Presentation only holds UIModels. Application wires the dependency graph at startup. Every boundary is a protocol — independently testable and swappable."

Then ask them (pick 1–2):

  • Is real-time sync a future requirement? That changes cache invalidation.
  • What does the existing analytics stack look like?
  • What's the biggest pain point in the current architecture that this redesign should address?
