Delta — What This Scenario Adds

Scenario extension of the generic iOS architecture. Read this first — it describes what's the same and what this problem adds on top.

Same as generic architecture

  • Clean Architecture + MVVM + UIKit
  • DTO → Mapper → Domain Model
  • FetchPolicy (.fresh / .cached / .strict) on all Repository reads
  • Typed Param structs on every UseCase
  • @MainActor on ViewModel — all state mutations on main thread
  • defer { isLoading = false } — guaranteed cleanup on success and failure
  • [weak self] in all closures
  • Coordinator-based navigation, manual init injection (no DI framework)
  • Mock-the-layer-below testing strategy
  • ThirdPartyDataSource pattern — wraps third-party SDKs; app never calls SDK directly
  • async/await for I/O; Combine for reactive binding to @Published state
  • Infrastructure layer (Gateway suffix) — Gateway trigger is cross-layer span, not SDK imports; single-layer SDKs wrap in their natural layer (DataSource or Service); Domain defines protocol; concrete in Infrastructure; nothing depends on Gateway except DI wiring in Application
  • External layer (outermost ring) — actual SDKs and OS frameworks; UIKit / SwiftUI / Combine need no wrapper (reactive/UI primitives used directly); all other SDKs always wrapped; wrapper placement scope-based: single-layer SDK → DataSource or Service, cross-layer SDK → Gateway in Infrastructure
  • Idempotency keys on mutations — client-generated UUID at Param call site for any retryable mutation
  • HTTP 409 ≠ 5xx — concurrency conflicts and transient server errors must never share a code path

No pagination. Stories are 24-hour bounded. The full metadata response is ~50 KB. A single getStoriesAfter(cursor:) endpoint handles both initial load (cursor = nil) and delta updates (cursor = latest known ID). Three pagination endpoints for 50 KB of data adds complexity with no benefit.
StoryOrderService is a Domain Service, not a UseCase. It is stateful — it holds a Set<Story.ID> of seen IDs that must survive across swipes and foreground re-entries. A stateless UseCase cannot hold that state.

Requirements

Functional

  • Full-screen story slideshow: swipe left/right or auto-advance every 10 seconds
  • Stories expire 24 hours after posting — expired stories must not be displayed
  • "Next" story = most recently posted unseen story (unseen-first, recency sort)
  • Once the last story is reached, loop back to the first (infinite scroll)
  • Offline mode: cached stories remain browsable without connectivity
  • Each story shows: photo, author avatar, author name, timestamp

Non-Functional

  • UI performance — no frame drops during swipe; no visible spinners mid-session
  • Network resilience — graceful degradation on slow or intermittent connections
  • Storage efficiency — avoid redundant image data; decoded UIImage cache, not raw bytes
  • Backend efficiency — 10-minute throttle guard; delta-only updates after initial load

API Design

Endpoint

GET /stories?after={photoID}
ParameterTypeDescription
afterInt?ID of the most recent locally known story. Omit for initial load.

Response: array of StoryDTO — newest-first, all unexpired stories posted after the cursor.

Two calls, one endpoint

Initial load
GET /stories (no cursor) — returns all stories from the last 24 hours
Delta update
GET /stories?after=3 — returns only stories posted after ID 3

Capacity estimate

100 bytes/field × 500 stories × ~0.25 compression × 4 string fields
  ≈ 50,000 bytes ≈ 50 KB

// Images are NOT in this response.
// API returns a URL string; binary is fetched lazily from Amazon S3.
Why not pagination? Classic cursor pagination (initialLoad(pageSize), headLoad, tailLoad) is designed for feeds of arbitrary depth. Stories are time-bounded. 50 KB fits in a single response. Pagination adds 3 endpoints and client-side merging complexity for zero benefit.

Mapper

struct StoryMapper {
    static func toDomain(_ dto: StoryDTO) -> Story? {
        guard
            let photoURL = URL(string: dto.photoURL),
            let avatarURL = URL(string: dto.profilePicURL)
        else { return nil }
        return Story(
            id: dto.photoID,
            photoURL: photoURL,
            profilePicURL: avatarURL,
            authorName: dto.authorName,
            createdAt: Date(timeIntervalSince1970: TimeInterval(dto.createdAt)),
            expireAt: Date(timeIntervalSince1970: TimeInterval(dto.expireAt))
        )
    }
}

Initial load

      → isLoading = true
      → FetchStoriesUseCase.execute(policy: .cached, param: FetchStoriesParam(cursor: nil))
          → StoryRepository.fetch(policy: .cached, param:)
              → StoryLocalDataSource.read() → [StoryDTO]? → StoryMapper → [Story]  // fast return if fresh
              → // if stale:
              → StoryRemoteDataSource.getStoriesAfter(cursor: nil) → [StoryDTO]
              → filter expireAt < now
              → StoryMapper.toDomain() → [Story]
              → StoryLocalDataSource.write([StoryDTO])
          → returns [Story]
      → StoryOrderService.ordered(stories:) → sorted [Story] (unseen-first)
      → ViewModel maps Story → StoryUIModel
      → @Published currentStory updated → ViewController renders
      → PrefetchStoryImageUseCase.execute(url: nextStory.photoURL)   // ViewModel → UseCase → StoryRepository → ImageSDKDataSource
      → defer: isLoading = false

Delta update (foreground re-entry)

applicationDidBecomeActive
  → StoryViewModel.refresh()
      → FetchStoriesUseCase.execute(policy: .cached, param: FetchStoriesParam(cursor: latestKnownID))

          → returns merged [Story]

Technical Deep Dives

1. Three-UIImageView Carousel (View Recycling)

Three UIImageView instances are laid out side by side. On each swipe, no new views are allocated — existing views are repositioned and their images swapped at rest:

// Before swipe →:  [ story N-1 | story N  | story N+1 ]  (N on screen)
// After  swipe →:  [ story N   | story N+1| story N+2 ]  (N+1 on screen)
//
// Leftmost view moved to rightmost position.
// Image swapped at rest. O(1) layout work per swipe.
Why not UICollectionView? UICollectionViewFlowLayout snap-scrolling makes seamless wrap-around (last → first story) awkward — you'd need to insert fake items or hack the content offset. Manual recycling handles the loop trivially.
Why not UIPageViewController? Same wrap-around problem, plus it instantiates new child view controllers per page — higher memory overhead for an infinite loop.

Gesture: UISwipeGestureRecognizer completes the transition automatically if >50% of screen width is covered. Short fast swipes (velocity-based) also trigger the flip.

2. SDWebImage as ImageSDKDataSource

LibraryWhat it cachesCost per display
NSURLCacheRaw HTTP response bytesbytes → Data → UIImage decode (CPU, potentially on main thread)

ThirdPartyDataSource pattern: ImageSDKDataSource implements ImageDataSourceProtocol. ViewController and ViewModel never import SDWebImage. Swapping the library = one file change.

3. Auto-Advance Timer

// ❌ Wrong — timer starts on transition; user sees spinner during their window
func showNext() {
    startAutoAdvanceTimer()
    imageView.sd_setImage(with: url)
}

// ✓ Correct — timer starts only after image is ready
func showNext() {
    imageSDKDataSource.load(url: url, into: imageView) { [weak self] in
        self?.startAutoAdvanceTimer()
    }
}

4. Story Expiry Enforcement

StoryRepository filters expireAt < Date() before returning Domain models on every cache read. A story cached 23 hours ago is not displayed if it has since expired. No separate background timer needed — the check runs on every read path.

5. 10-Minute Throttle Guard

StoryLocalDataSource records the timestamp of the last successful fetch. StoryRepository checks it when applying FetchPolicy.cached:

// In StoryRepository.fetch(policy: .cached, param:)
let age = Date().timeIntervalSince(localDataSource.lastFetchedAt)
if age < 600 {
    return localDataSource.read()   // skip network
}
let dtos = try await remoteDataSource.getStoriesAfter(cursor: param.cursor)
// ... save + return
The ViewModel never makes the time decision. It always uses FetchPolicy.cached. The throttle logic lives entirely in the Repository — one place to change it.

6. Pre-fetching

StoryViewModel calls PrefetchStoryImageUseCase.execute(url: nextStory.photoURL) immediately after rendering the current story. The UseCase delegates to StoryRepository.prefetchImage(url:) → ImageSDKDataSource.prefetch(url:). By the time the user swipes, the next image is already decoded in SDWebImage's memory cache — zero visible latency, no spinner. ViewModel never calls the DataSource directly (no Presentation → Data bypass).

Cancel prefetch on backward swipe. If the user swipes back to story N-1, the in-flight prefetch for story N+1 should be cancelled. Stale prefetch requests waste bandwidth and pollute the cache.

7. Viewed State & Delta Merge

StoryOrderService holds a Set<Story.ID> in memory. On delta merge:

  1. New stories are added to the pool
  2. ordered(stories:) runs on the full merged pool
  3. Stories with IDs in the seen set are sorted to the back
  4. next(in:) returns the first unseen story
Open question: Whether the seen-set is persisted to disk (CoreData/JSON) alongside story metadata or is memory-only. If memory-only, the user re-sees all stories on next app launch — intentional for stories (Instagram does this) but worth raising in an interview.

Common Pitfalls

Starting the auto-advance timer on transition. Burns the user's 10-second window on a loading spinner. Timer must start only after ImageSDKDataSource signals download completion.
Using NSURLCache for images. Re-decodes bytes → UIImage on every cache hit. Under fast swiping this causes visible frame drops. Use SDWebImage (decoded UIImage cache).
Polling on every viewWillAppear. Without the 10-minute throttle guard in the Repository, every tab switch hits the backend. Always apply FetchPolicy.cached on foreground re-entry.
Allocating new UIImageViews per swipe. Triggers layout passes on every navigation. Always reuse the three existing views — move and swap, never allocate.
Not filtering expireAt on cache read. A story cached hours ago may have expired before display. Filter at the Repository level on every read, not just on fetch.
Not carrying viewed-state through delta merge. If StoryOrderService drops the seen set on rebuild, previously-viewed stories reappear at the front after a refresh.

Key Takeaways

Single delta endpoint, not pagination. Stories are 24-hour bounded — the full metadata response is ~50 KB. getStoriesAfter(cursor:) handles both initial load and delta updates. Pagination adds 3 endpoints and merging complexity for zero benefit.
StoryOrderService is a Domain Service, not a UseCase. It holds stateful viewed-state across swipes and foreground re-entries. A UseCase executes once and returns; a Domain Service persists.
10-minute throttle guard lives entirely in StoryRepository. The ViewModel always passes FetchPolicy.cached. The time-based decision is the Repository's concern — never the ViewModel's.
Three recycled UIImageViews for seamless infinite loop. UICollectionView and UIPageViewController both struggle with wrap-around. Manual recycling is O(1) layout work per swipe.
SDWebImage as a ThirdPartyDataSource. Caches decoded UIImage — zero CPU decode overhead on swipe. App calls ImageDataSourceProtocol; SDK never leaks out of the DataSource layer.
Auto-advance timer starts after download, not transition. Guarantees a full 10 seconds of visible content. SDWebImage completion callback → start timer.
