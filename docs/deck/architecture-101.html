Architecture 101

The Layers

ViewController
Layout, user events, binds to ViewModel via Combine
↓
ViewModel
@MainActor · @Published state · owns loading + error
↓
UseCase
Business rules, validation, orchestration (async let / TaskGroup)
↓
Repository
Translates Param → Request · owns Domain↔Data boundary
↓
DataSource
Builds URL · calls APIClient · returns DTO
↓
APIClient
URLSession wrapper · GET / POST / PUT / PATCH / DELETE
Layer rule: Presentation → Domain ← Data. Domain depends on nothing.

Data Flow — Read

ViewModel.load()
  → UseCase.execute(policy: .fresh, param: SearchTracksParam)
    → Repository.searchTracks(policy:param:)
      → TrackSearchRequest(query:offset:limit:)  ← Param → Request
      → DataSource.searchTracks(request)
        → APIClient.get(url)               ← URLSession
          → iTunesSearchResponse (DTO)
      → TrackMapper.toDomain(dto) → Track  ← only Mapper knows both
    → [Track]
  → TrackUIModelMapper.toUIModel(track) → TrackUIModel
→ ViewModel.tracks = [TrackUIModel]     ← VC never sees Track

Data Flow — Write

ViewModel.createPlaylist(name:)
  → UseCase.execute(param: CreatePlaylistParam)
    → guard !name.isEmpty else throw PlaylistError.emptyName
    → Repository.createPlaylist(param:)
      → CreatePlaylistRequest(name:description:trackIds:)
      → DataSource.createPlaylist(request)
        → APIClient.post(url, body: request)   ← Encodable, id excluded
          → PlaylistDTO (response)
      → PlaylistMapper.toDomain(dto) → Playlist
→ ViewModel.playlist = PlaylistUIModel
Rule: No FetchPolicy on writes — always hit the network. FetchPolicy only answers "where do I read from?"

Key Types

Param<Query, Path>

struct Param<Query, Path> {
    let query: Query   // filter / body fields
    let path: Path     // URL path segments
}
typealias SearchTracksParam = Param<SearchTracksQuery, Void>
typealias UpdatePlaylistParam = Param<UpdatePlaylistQuery, UpdatePlaylistPath>

Adding a field to Query or Path never breaks call sites.

FetchPolicy

static let fresh  = FetchPolicy(force: true,  allowStale: false) // always network
static let cached = FetchPolicy(force: false, allowStale: true)  // cache-first
static let strict = FetchPolicy(force: false, allowStale: false) // cache only, fail if missing

Travels from ViewModel → UseCase → Repository. Read UseCases only.

Request Structs (Data layer)

struct TrackSearchRequest { let query: String; let offset: Int; let limit: Int }
struct UpdatePlaylistRequest: Encodable {
    let id: Int          // path param — excluded via CodingKeys
    let name: String
    enum CodingKeys: String, CodingKey { case name }
}

UIModel + UIModelMapper

enum TrackUIModelMapper {
    static func toUIModel(_ track: Track) -> TrackUIModel {
        let seconds = track.durationMs / 1000
        TrackUIModel(duration: String(format: "%d:%02d", seconds / 60, seconds % 60), ...)
    }
}

ViewController only ever sees UIModel. Domain entities never reach the ViewController.

Concurrency Patterns

2 concurrent fetches → async let

async let tracks = trackRepository.searchTracks(policy: policy, param: ...)
async let playlists = playlistRepository.fetchPlaylists(policy: policy)
return HomeData(featuredTracks: try await tracks, playlists: try await playlists)

N concurrent fetches → withThrowingTaskGroup

return try await withThrowingTaskGroup(of: HomeSection.self) { group in
    for (genre, query) in param.query.genreQueries {
        group.addTask { try await repository.searchTracks(policy: policy, param: ...) }
    }
    var sections: [HomeSection] = []
    for try await section in group { sections.append(section) }
    return sections.sorted { ... }  // order is non-deterministic — always sort
}

ViewModel state pattern

@MainActor final class TrackListViewModel {
    @Published private(set) var tracks: [TrackUIModel] = []
    @Published private(set) var isLoading = false
    @Published private(set) var errorMessage: String?

    func load() {
        isLoading = true
        Task {
            defer { isLoading = false }   // guaranteed on success AND failure
            do {
                tracks = try await useCase.execute(...)
            } catch {
                errorMessage = error.localizedDescription
            }
        }
    }
}

Trade-off Talking Points

Repository + DataSource vs Repository alone?

Repository owns the Domain↔Data boundary (Param→Request, DTO→Domain). DataSource owns HTTP details. Splitting them means you can swap clients or add caching without touching business mapping logic.

Param<Query, Path> vs plain arguments?

Adding a field doesn't break every call site. Path and Query have different semantic roles — typed struct makes intent explicit, type system enforces it.

No BaseUseCase protocol?

Read and write have different contracts — FetchPolicy vs none. A base type either lies about the contract or recreates what per-UseCase protocols already do with less ceremony.

UIModelMapper as a separate type?

Only when it has display formatting logic worth testing independently. A pure 1:1 field copy doesn't need a dedicated type — inline in ViewModel is cleaner. Consistency for its own sake is boilerplate.

@MainActor on ViewModel?

All @Published mutations must happen on the main thread. @MainActor enforces this at compile time instead of relying on manual DispatchQueue.main.async.

Why the Rule Exists

The real insight behind Clean Architecture is about rate of change.

LayerChangesExample trigger
DomainSlowlyA business rule changes
DataFrequentlyServer renames a JSON field
PresentationVery frequentlyDesign gets iterated
Volatile depends on stable. Stable depends on nothing.

The dependency rule forces volatile layers to depend on stable ones, never the reverse. If Domain depended on Data, a server-side rename would force changes in your business logic — which is absurd.

The Plugin Metaphor

Think of Domain as a standalone business application. Everything else snaps onto it as a plugin.

PluginWhat it means
UI is a pluginSwap UIKit for SwiftUI without changing any business logic
Database is a pluginSwap URLSession for Alamofire, or REST for GraphQL, without touching UseCases
Framework is a pluginUIKit, Combine, CoreData — implementation details, not load-bearing architecture

This is why RepositoryProtocol is defined in Domain, implemented in Data. Domain defines the interface it needs. Data conforms to it. The direction of the source code dependency is the opposite of the runtime call direction. This reversal is called Dependency Inversion — the D in SOLID.

At runtime: ViewModel → UseCase → Repository protocol → concrete Repository (networking). The business logic has no idea URLSession exists.

Testability Is a Consequence

People describe Clean Architecture as "good for testability" as if testability were the point. It isn't. Testability is a consequence of proper dependency direction, not the motivation for it.

The real reason you can test a ViewModel with a mock repository: the ViewModel literally cannot import the real Repository without violating the dependency rule. The architecture forces you into testable seams.

Diagnostic: If testing a component requires mocking ten things, the component is doing too much — it has absorbed responsibilities from multiple layers. Test difficulty is a signal, not a test infrastructure problem.

How to Place Any Piece of Logic

When you don't know where something belongs, ask:

QuestionAnswer → Layer
Does this change when the API shape changes?Data layer
Does this change when the design changes?Presentation layer
Is this a business rule true regardless of infrastructure?Domain layer

Three-second version: "Domain owns the contracts. Data fulfills them. Presentation consumes them. None of the outer layers know about each other."

Cost of Breaking the Rule

Each violation creates a hidden coupling that only reveals itself when something changes.

ViolationWhen it hurts
ViewModel imports DTORenaming a JSON field requires a ViewModel change
UseCase uses URLSession directlyCan't test the UseCase without a live network
Repository constructs a Domain model from raw JSONAPI change breaks business logic
ViewController reads from a RepositoryNavigation breaks when the data layer is refactored
Compound effect: once a few violations exist, the architecture degrades rapidly — new code copies the pattern it sees. A single DTO import in a ViewModel becomes "that's just how we do it here" within weeks.

Combine vs async/await — Trade-offs

StrengthsWhen to reach for it
async/awaitReadable, structured cancellation, no AnyCancellable lifecycle to manageSequential async work, one-shot requests, clearer error propagation
CombineDeclarative stream composition, operators (debounce, removeDuplicates, combineLatest)Multiple subscribers reacting to shared state, UI binding, complex transformations

They compose well — a Task { } inside a sink is valid when you need async work triggered by a publisher. Many codebases use both.

Interview answer: "I'd use whichever the team is already using consistently. If starting fresh, I'd lean toward async/await for data work and Combine for reactive binding — but both are valid, and the most important thing is not mixing them for the same operation without a clear reason."

Algorithm Complexity — O Notation

NotationMeaningSwift example
O(1)Constant — doesn't grow with inputDictionary lookup by key
O(n)Linear — one passfilter, map, forEach, Dictionary(grouping:)
O(n log n)Log-linear — sortingsorted, sort
O(n²)Quadratic — nested loopsNaive duplicate check — avoid, replace with a Set
Chaining rule: when operations are chained, the slowest one dominates. filter O(n) → sort O(n log n) = total O(n log n).

What to say: "My approach is X — that's O(?) because Y. If the dataset were much larger I'd consider Z." Always name the complexity before you write the code.
