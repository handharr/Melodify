Delta — What This Scenario Adds

Scenario extension of the generic iOS architecture. Read this first — it describes what's the same and what this problem adds on top.

Same as generic architecture

  • Clean Architecture + MVVM + UIKit
  • DTO → Mapper → Domain Model
  • FetchPolicy (.fresh / .cached / .strict) on all Repository reads
  • Typed Param structs on every UseCase
  • @MainActor on ViewModel — all state mutations on main thread
  • defer { isLoading = false } — guaranteed cleanup on success and failure
  • [weak self] in all closures
  • Coordinator-based navigation, manual init injection (no DI framework)
  • Mock-the-layer-below testing strategy
  • ThirdPartyDataSource pattern — wraps third-party SDKs; app never calls SDK directly
  • async/await for I/O; Combine for reactive binding to @Published state
  • Idempotency keys on mutations — client-generated UUID at Param call site for any retryable mutation
  • HTTP 409 ≠ 5xx — concurrency conflicts and transient server errors must never share a code path
  • Infrastructure layer (Gateway suffix) — Domain defines protocol; concrete in Infrastructure; nothing depends on Gateway except DI wiring in Application

Key decisions unique to this scenario

Cursor-based pagination over offset. New posts arrive continuously. Offset pagination drifts — page 2 after an insertion returns one item you already saw. Cursor anchors to postID, immune to insertions above it.
LikeService must be app-scoped. It is stateful (holds a durable retry queue) and long-lived — must survive foreground/background cycles. A stateless UseCase cannot hold or retry queued actions. Register at AppCoordinator, same as PlayerService.
Optimistic UI with Core Data-backed queue. The like queue (successfullySent: false) is persisted — not in-memory. An app kill mid-retry does not lose the user's intent. On relaunch, LikeService reads pending Like records and resumes retrying.
SDWebImage as ImageSDKDataSource. URLCache stores raw Data bytes — every display requires UIImage(data:), a CPU decode hit. SDWebImage caches the already-decoded UIImage in memory + disk. Repeat displays are zero-cost.
ViewModel implements UICollectionViewDataSource. The ViewModel already owns items: [FeedCellUIModel]. It is the natural place to answer "how many cells?" and "which model at index N?". Avoids a separate data source object with shared mutable state.
Feed freshness extends FetchPolicy.cached. FetchPolicy.cached says "use local if available". This scenario adds a time constraint — local is only valid if fetchedAt < 2 minutes ago. The Repository checks the timestamp before deciding whether .cached qualifies.

API Design

Strategy: REST + cursor-based pagination. Authentication assumed.

Why cursor over offset? New posts arrive continuously. If 5 posts are inserted while the user is on page 2, offset page 3 now overlaps with page 2 — the user sees duplicates. Cursor anchors to postID, immune to insertions above the anchor.
Why batch likes? A user who likes several posts while offline sends one request on connectivity restore instead of N. Reduces reconnect burst load.
Image URLs, not binary data. PostDTO.photoURLs contains S3 URLs. The API stays lightweight; ImageSDKDataSource resolves images independently.

Endpoints

GET /users/<userID>/feed
  Params:
    postID  — cursor (ID of last seen post; omit for first page)
    limit   — Int (20)
    page    — "prev" | "next"
  Response:
    posts: [PostDTO]
    nextCursor: String?

POST /users/<userID>/likes
  Body:
    [{ postID: Int, value: Bool }]
  Response: 204 No Content

struct PostDTO: Codable {
    let postID: Int
    let type: Int               // 0 = photo, 1 = album
    let authorID: Int
    let likeCount: Int
    let caption: String
    let location: String?
    let photoURLs: [String]

UIModels (Presentation layer only)

enum FeedCellUIModel {
    case photo(PhotoCellUIModel)
    case album(AlbumCellUIModel)
}

struct PhotoCellUIModel {
    let title: String
    let avatarURL: URL
    let imageURL: URL           // single photo
    let likeCount: Int
    var isLiked: Bool
}

struct AlbumCellUIModel {
    let title: String
    let avatarURL: URL
    let imageURLs: [URL]        // carousel
    let likeCount: Int
    var isLiked: Bool
}
Why FeedCellUIModel is an enum. A single Post-like struct with optional fields forces cells to defensive-unwrap. An enum encodes the variant at the type level — the compiler enforces exhaustive handling. Cells never touch the domain layer.

Dependency Injection (Composition Root)

No DI framework. Dependencies flow inward via init. AppCoordinator owns the composition root and holds app-scoped objects as strong properties.

// AppCoordinator — composition root
let client       = APIClient()

// Data layer
let feedRemoteDS = FeedRemoteDataSource(client: client)
let likeRemoteDS = LikeRemoteDataSource(client: client)
let feedLocalDS  = FeedLocalDataSource()              // wraps Core Data

// Repositories
let newsFeedRepo = NewsFeedRepository(remote: feedRemoteDS, local: feedLocalDS)
let likeRepo     = LikeRepository(remote: likeRemoteDS, local: feedLocalDS)

// Domain Service — strong property on AppCoordinator; survives screen transitions
let likeService  = LikeService(repository: likeRepo)
likeService.resumePendingRetries()  // re-queue successfullySent=false on launch

// UseCases — stateless; created per-navigation
let fetchFeedUseCase = FetchFeedUseCase(repository: newsFeedRepo)

// Presentation — Coordinator creates this when pushing the feed screen
let viewModel = FeedViewModel(
    fetchFeed: fetchFeedUseCase,
    likeService: likeService
)
LikeService is a stored property on AppCoordinator — not created inside FeedViewController. This is what keeps it alive across screen transitions and app lifecycle events. If owned by the ViewController it deallocates on pop and the retry queue is lost.

Data Flow

1. Feed Load Flow

FeedViewController.viewDidLoad()
    → FeedViewModel.load()
        → isLoading = true
        → FetchFeedUseCase.execute(policy: .cached, param: FeedParam(userID:))
            → NewsFeedRepository.fetch(policy: .cached, param:)
                → FeedLocalDataSource.fetchedAt → if < 2 min → return cached DTOs  // instant
                → else: FeedRemoteDataSource.fetch()
                         → GET /users/<id>/feed?postID=<cursor>&limit=20
                         → [PostDTO]
                         → PostMapper.toDomain(dto) → [Post]
                         → FeedLocalDataSource.save(dtos)  // write-through cache
        → FeedViewModel maps [Post] → [FeedCellUIModel]
        → @Published items updated → UICollectionView reloads
        → defer: isLoading = false

2. Like Flow (Optimistic, Offline-Safe)

User taps like at index i
    → FeedViewModel.toggleLike(at: i)
        → items[i] optimistically mutated (toggle isLiked, ±1 likeCount)  // instant UI
        → LikeService.enqueue(Like { userID, postID, successfullySent: false })
            → FeedLocalDataSource.save(Like)  // persisted to Core Data
            → LikeRemoteDataSource.post([{ postID, value }])
                → success:  FeedLocalDataSource.update(Like, successfullySent: true)
                → failure:  leave successfullySent = false
                    → background retry loop on connectivity restore

3. Scroll Prefetch Flow

UICollectionView triggers prefetchItemsAt(indexPaths:)
    → FeedViewModel.prefetch(at: indexPaths)
        → start background data task per IndexPath (if not already fetching)

UICollectionView triggers cancelPrefetchingForItemsAt(indexPaths:)
    → FeedViewModel.cancelPrefetch(at: indexPaths)
        → cancel pending tasks — free bandwidth for visible cells

Cell.prepareForReuse()
    → imageView.sd_cancelCurrentImageLoad()   // cancel stale SDWebImage request

Deep Dives

1. Cursor-Based Pagination

Offset (breaks on live feed)

Page 1:  [A B C D E F G H I J]  → request page 2 (offset=10)
                   ↑
         New post "X" inserted here

Page 2 (offset=10):  [F G H I J K ...]  ← F already seen — duplicate!

Cursor (stable)

Page 1:  [A B C D E F G H I J]  → nextCursor = "J"
                   ↑
         New post "X" inserted here

Page 2 (cursor="J"):  [K L M ...]  ← starts after J, unaffected
The cursor is opaque. The client echoes it back verbatim — never parses or constructs it. The server decides what's inside (ID, timestamp, composite key). Decouples client from server internals — same principle as treating an HTTP ETag as opaque.
CursorOffset
Live-updating feed✅ stable❌ skips / duplicates
Static dataOverkill✅ simpler
Jump to arbitrary page❌✅
Server complexityHigherSimple
Rule: cursor when data moves (live feed, synced library), offset when it doesn't (search results, static lists).

2. SDWebImage vs URLCache

URLCacheSDWebImage
StoresRaw Data (HTTP bytes)Decoded UIImage object
Display costUIImage(data:) every displayZero — object already decoded
ThreadDecode on main thread (risk)Background decode, main thread display
Disk cacheYes (HTTP response)Yes (decoded image file)
Memory cacheNSURLCache LRUNSCache (auto-evict under memory pressure)
SDWebImage as ImageSDKDataSource: ImageDataSourceProtocol. The rest of the app calls the protocol — never SDWebImage directly. Swapping the library = one file change in one DataSource.
Common mistake: using URLCache for images because it's "built-in". It caches raw bytes — you pay UIImage(data:) decode on every cell display. On a fast-scrolling feed with large photos this is a measurable CPU hit.

3. Resource Starvation Mitigation

When a user flicks through the feed rapidly, hundreds of image requests queue for cells that are no longer visible. Visible cells starve for bandwidth.

1. Cancel prefetch tasks
Implement cancelPrefetchingForItemsAt. Stop data tasks for IndexPaths no longer approaching the viewport. Without this, stale requests pile up.
2. Cancel on cell reuse
prepareForReuse calls sd_cancelCurrentImageLoad(). Prevents a recycled cell from showing the previous post's image briefly while the new request loads.
3. Defer until scroll ends
Optionally, only fire new data tasks in scrollViewDidEndDecelerating. Bandwidth goes entirely to what the user is actually reading during rapid scroll.
Always pair prefetchItemsAt with cancelPrefetchingForItemsAt. Implementing only the prefetch half is worse than implementing neither — you get all the queuing with none of the cleanup.

4. Feed Freshness Policy

The generic architecture's FetchPolicy.cached returns local data if any exists. This scenario adds a time constraint.

// Inside NewsFeedRepository.fetch(policy:param:)
if policy == .cached {
    let age = Date().timeIntervalSince(localDataSource.fetchedAt ?? .distantPast)
    if age < 120 { return localDataSource.fetchAll() }  // serve from cache
}
// else: fall through to network
Where to store fetchedAt: a fetchedAt: Date field on a FeedMetadata Core Data entity (co-located with the cached posts), or a UserDefaults key. Either works. UserDefaults is simpler; Core Data keeps it transactional with the cached data.
2 minutes is a product decision. Too short → redundant API calls on quick re-navigations. Too long → stale feed. Expose as a configurable constant; don't hardcode deep in the Repository.

5. Optimistic UI + Durable Like Queue

The generic mutation flow: ViewModel → UseCase → Repository → server, then update UI on response. This scenario inverts the order for likes: UI updates first, server is eventual.

// Generic mutation flow (awaits server)
ViewModel → UseCase → Repository → server → ✅ → update @Published state

// Optimistic flow (UI first)
ViewModel → mutate items[i] immediately  // user sees change instantly
          → LikeService.enqueue(Like { ..., successfullySent: false })
              → persist to Core Data
              → attempt network  →  success: successfullySent = true, delete record
                                 →  failure: retry on connectivity restore
The queue is durable — not session-scoped. On app relaunch, LikeService reads all Like records with successfullySent = false from Core Data and resumes retrying. An app kill mid-retry does not lose the user's intent.
Records must be cleaned up. The retry loop must set successfullySent = true and eventually delete the record on confirmed success — otherwise the Core Data queue grows unbounded over time.
LikeService must not be owned by a ViewController. If scoped to a screen, it deallocates when the user navigates away — the retry queue disappears. Register at AppCoordinator level.

Pitfalls / Gotchas

Using URLCache for images. It stores raw bytes — you decode on every display. Wrap SDWebImage in ImageSDKDataSource instead.
Offset pagination on a live feed. Insertions at the top shift offsets — duplicates appear. Use cursor-based pagination.
Not implementing cancelPrefetchingForItemsAt. Stale requests pile up and starve visible cells. Always pair prefetch with its cancel counterpart.
Forgetting sd_cancelCurrentImageLoad() in prepareForReuse. Fast scrollers see wrong images briefly — the recycled cell shows the previous request's result.
successfullySent records never cleaned up. The retry loop must delete the record on confirmed success, or the Core Data queue grows unbounded.
LikeService scoped to a ViewController. It deallocates on pop — the retry queue disappears. Register at AppCoordinator level, same as PlayerService.
Creating LikeService inside a ViewController instead of the composition root. If FeedViewController owns it, the service deallocates on pop — the retry queue is lost. It must be a stored property on AppCoordinator, passed down via init injection.

Interviewer Talking Points

On pagination: "I'd use cursor-based pagination — it's stable when new posts arrive, unlike offset which drifts and causes duplicates or skips."
On optimistic UI: "The Like domain model carries a successfullySent flag. LikeService updates the UI optimistically and retries in the background until the flag flips to true. The queue is persisted in Core Data — an app kill doesn't lose the intent."
On image caching: "For images I'd wrap SDWebImage in an ImageSDKDataSource. URLCache stores raw bytes and you pay the decode hit every time. SDWebImage caches the decoded UIImage, so repeat displays are zero-cost."
On scroll performance: "I'd use UICollectionView with UICollectionViewDataSourcePrefetching. That protocol gives me cancelPrefetchingForItemsAt — without it, rapid scrolling queues hundreds of stale requests and starves visible cells."
On cell design: "The ViewModel maps Post domain objects into typed FeedCellUIModel variants before the collection view sees them. Cells are dumb renderers — they never touch the domain layer."
On offline: "Core Data for metadata, S3 URLs (not raw bytes) in the API response, SDWebImage for image disk cache — these three together mean the feed works fully offline after the first load."
On freshness: "Feed freshness extends FetchPolicy.cached with a 2-minute timestamp at the Repository level. If fetchedAt is recent, skip the network. The Repository is the only place that reads this timestamp."

Open Questions

  • Retry policy for failed likes: Back-off strategy, retry cap, and user-facing error after exhausting retries. Suggested position: 3 attempts with exponential back-off, then surface a non-blocking error banner.
  • Exact prefetch trigger threshold: How many cells from the bottom triggers the next cursor page request. Product decision — e.g., last 5 cells ≈ 25% of one page.
  • Feed refresh UI transition: Full reload vs NSDiffableDataSourceSnapshot diff on cache expiry. Diff avoids flash; full reload is simpler.
  • Cache timestamp storage: fetchedAt as a Core Data entity field vs UserDefaults keyed by userID. Either works; UserDefaults is simpler.
  • Network restoration trigger: NWPathMonitor (modern, built-in) vs Reachability (third-party, legacy) for flushing the pending like queue. Prefer NWPathMonitor.
