Delta — What This Scenario Adds

Scenario extension of the generic iOS architecture. Read this first — it describes what is the same and what this problem adds on top.

Same as generic architecture

  • Clean Architecture + MVVM + UIKit
  • DTO → Mapper → Domain Model
  • FetchPolicy (.fresh / .cached / .strict) on all Repository reads
  • Typed Param structs on every UseCase
  • @MainActor on ViewModel — all state mutations on main thread
  • defer { isLoading = false } — guaranteed cleanup on success and failure
  • [weak self] in all closures to avoid retain cycles
  • Coordinator-based navigation, manual init injection (no DI framework)
  • Mock-the-layer-below testing strategy
  • ThirdPartyDataSource facade — app calls protocol, never SDK directly
  • Idempotency keys on mutations — UUID generated at Param call site
  • HTTP 409 ≠ 5xx — concurrency conflicts and transient server errors must never share a code path
  • Infrastructure layer (Gateway suffix) — Domain defines protocol; concrete in Infrastructure; nothing depends on Gateway except DI wiring in Application

Key decisions unique to this scenario

SSE over polling. A 1-second poll for 10 minutes = 600 HTTP requests per order. SSE replaces all of that with one persistent connection. Polling is simpler but wastes battery and overloads the backend.
SSE over WebSockets. Courier tracking is unidirectional (server → client only). WebSockets add bidirectional complexity this scenario doesn't need.
Server-hosted basket. Cross-device continuity — start on iPhone, finish on web. Trade-off: every mutation requires a network round-trip.
OrderService must close SSE on screen exit. If the stream stays open after the Order Status screen is dismissed, it drains battery and holds a server connection. Close in viewDidDisappear equivalent.
UIImageView extension isolates the image library. Same principle as ThirdPartyDataSource — call sites call imageView.setImage(url:), never SDWebImage directly. Swapping libraries touches one file.

Data Model

// Domain models — pure Swift structs, no UIKit or Foundation networking types

struct User {
    userID: Int
    name: String
    email: String
    addresses: [Address]
    lastUsedAddress: Address   // drives initial restaurant list without extra fetch
}

struct Address {
    addressID: Int
    label: String
    latitude: Double
    longitude: Double
    // + city, street, flat, postcode
}

struct Restaurant {
    restaurantID: Int
    name: String
    rating: Int
    address: Address
    imageURL: URL
}

struct Dish {
    dishID: Int
    restaurantID: Int    // FK: one-to-many with Restaurant
    name: String
    price: Double
    imageURL: URL
}

struct Basket {
    basketID: Int
    userID: Int
    restaurantID: Int
    selectedDishes: [(dishID: Int, count: Int)]
}

struct Order {
    orderID: Int
    status: OrderStatus
    basket: Basket
    courierLatitude: Double    // updated via SSE events
    courierLongitude: Double
}

enum OrderStatus {
    case placed, preparing, pickedUp, inTransit, delivered, cancelled
}
Why OrderStatus enum over String? The video uses status: String. An enum makes status transitions exhaustive at compile time — adding a new state is a compile error everywhere it's handled. Strings let invalid states slip through silently.
Why lastUsedAddress on User? The Restaurant List screen needs an address to filter restaurants. Storing the last-used address on the User model means the initial restaurant list can load immediately on app launch without an extra address-selection step. It's a UX shortcut that avoids an extra screen on the happy path.

API Design

REST Endpoints

// Restaurants & Dishes
GET   /users/<userID>               → UserDTO
GET   /restaurants/<addressID>      → [RestaurantDTO]
GET   /dishes/<restaurantID>        → [DishDTO]

// Basket
POST  /basket  {userID, restaurantID, dishID, count}  → BasketDTO
PATCH /basket  {basketID, dishID, count}              → BasketDTO
GET   /basket/<basketID>            → BasketDTO

// Orders
POST  /orders  {userID, basketID, idempotencyKey: UUID}  → OrderDTO
GET   /orders/<orderID>             → OrderDTO

// SSE — real-time courier tracking
GET   /live-order-status/<orderID>  → text/event-stream
Idempotency on POST /orders. A network timeout after the server succeeds causes the client to retry, creating a duplicate order. Include a client-generated UUID (idempotencyKey) — the server returns the existing record on duplicate submission. Generate the key at the CreateOrderParam call site, not inside the Repository.
Why SSE over polling for /live-order-status? Polling at 1-second intervals over a 10-minute order = 600 HTTP requests, most returning no new data. SSE is one persistent connection — the server writes only when state changes. Battery-efficient and backend-friendly.

Architecture

Pattern: MVVM + Clean Architecture + UIKit — Presentation → Domain ← Data.

Why manual DI over Swinject (the video's choice)? Swinject adds a framework dependency and can crash at runtime if a registration is missing — the failure surfaces only at first use, not at boot. Manual init injection via Coordinator exposes all dependencies at compile time. The Coordinator is the composition root; it's also the DI container — no extra framework needed.

Data Flow

Restaurant List Screen

Pattern A — two awaits in the ViewModel. async/await returns once; a single execute() cannot update state twice.

AppCoordinator reads User.lastUsedAddress from UserRepository (.cached)

RestaurantListViewModel.load()
    → isLoading = true

    // Phase 1 — cache (instant)
    if let cached = try? await FetchRestaurantsUseCase.execute(policy: .strict, param: FetchRestaurantsParam(addressID:))

        → ViewModel maps [Restaurant] → [RestaurantUIModel]
        → @Published restaurantList updated → UI renders immediately

    // Phase 2 — network (background)

        → DTO → Mapper → [Restaurant]
        → RestaurantLocalDataSource.save(dtos)  // Core Data updated
        → ViewModel maps [Restaurant] → [RestaurantUIModel]
        → @Published restaurantList updated → ViewController re-renders

    → defer: isLoading = false

Order Status Screen — SSE Flow

// Screen appears
OrderStatusViewController.viewDidAppear()
  → OrderStatusViewModel.startTracking(orderID:)
      → OrderService.startTracking(orderID:)
          → orderSSEGateway.stream(orderID:)     // OrderSSEGatewayProtocol (Infrastructure) → AsyncStream<OrderSSEEventDTO>
          → for await event in stream
              → OrderMapper.toDomain(event) → Order
              → publishes via AnyPublisher<Order, AppError>
      → ViewModel.orderUpdates sink → maps Order → OrderUIModel
      → @Published courierLocation + orderStatus → map pin moves, label updates

// Screen exits
OrderStatusViewController.viewDidDisappear()
  → OrderStatusViewModel.stopTracking()
      → OrderService.stopTracking()    // SSE connection closed immediately
Always close SSE on screen exit. A stream left open after navigation drains battery and holds a server connection with no consumer. stopTracking() must be called in every exit path — pop, dismiss, and app background.

Deep Dives

1. SSE vs Polling vs WebSockets

Why polling is wrong here

// 1-second polling over a 10-minute order
t=0s    GET /orders/{id} → "preparing"   ← wasted request
t=1s    GET /orders/{id} → "preparing"   ← wasted request
...
t=47s   GET /orders/{id} → "picked_up"   ← finally useful
// 600 requests total, 599 returned unchanged data

// SSE — one connection
t=0s    GET /live-order-status/{id}         ← persistent connection opened
t=47s   ← server: {status: "picked_up",  lat: 1.30, lng: 103.8}
t=89s   ← server: {status: "in_transit", lat: 1.31, lng: 103.81}
SSEPollingWebSocket
DirectionServer → client onlyClient pullsBidirectional
ProtocolHTTP/1.1 or HTTP/2HTTP/1.1ws://
Battery costLowHighLow
ComplexitySimpleSimplestHigher
Right for courier tracking?✅❌Overkill
Rule: Use SSE when data flows server → client only and you want push without bidirectional complexity. Use WebSocket when the client also needs to push — e.g. chat, collaborative editing.

2. OrderService — Scoping

Unlike PlayerService in a music app (app-scoped to keep audio alive across all screens), OrderService should be scoped to OrderCoordinator — created when the order flow starts, destroyed when the coordinator deallocates.

ScopeResult
App-scoped (too broad)SSE connection stays open between orders; wastes battery and server resources when not on Order Status screen
ViewController-scoped (too narrow)Service destroyed if VC is re-created; stream drops mid-tracking
Coordinator-scoped ✅Lives exactly as long as the order flow; created by OrderCoordinator, destroyed when coordinator pops
Compare to PlayerService. Music playback must continue when the user navigates away from the player screen — app-scoped is required. Courier tracking has no reason to continue once the Order Status screen is dismissed — coordinator-scoped is correct.

3. Basket Cache Strategy

The basket lives on the backend (primary source of truth) but is mirrored in BasketLocalDataSource (Core Data). This serves two purposes:

Instant restore
If the app is killed and relaunched mid-basket, the local cache shows the last known state immediately while a GET /basket fetch runs in the background.
Offline display
FetchPolicy.cached on FetchBasketUseCase returns the local cache first. The Repository writes network responses back to the cache when they arrive.
// BasketRepository — applies FetchPolicy
func fetchBasket(policy: FetchPolicy) async throws → Basket {
    switch policy {
    case .cached:
        if let cached = try localDataSource.fetch() {
            return BasketMapper.toDomain(cached)!
        }
        fallthrough
    case .fresh:
        let dto = try await remoteDataSource.fetch()
        try localDataSource.save(dto)
        return BasketMapper.toDomain(dto)!
    case .strict:
        guard let cached = try localDataSource.fetch() else { throw AppError.cacheMiss }
        return BasketMapper.toDomain(cached)!
    }
}

Common Pitfalls

Not closing SSE on screen exit. Stream stays open, drains battery, holds a server connection with no consumer. Always call stopTracking() in viewDidDisappear — not just on explicit back-tap.
ViewModel must not hold a reference to its View. This is the core MVVM contract — ViewController subscribes to @Published state via Combine, ViewModel has no back-reference. The MVP pattern (weak var view: ViewProtocol?) creates two-way coupling that makes the ViewModel untestable without a mock View.
UseCases and Domain Services must not call networking directly. Networking belongs in RemoteDataSource. The correct chain is UseCase → Repository → RemoteDataSource → APIClient. A service that calls a network client directly bypasses the Repository abstraction and couples the Domain layer to Data.
No idempotency key on order creation. A network timeout after the server succeeds causes a duplicate order on retry. Generate a UUID at the CreateOrderParam call site and include it in the POST body.
Swinject missing registration crash. Registrations missing at startup crash at first use — hard to catch in tests. Manual init injection via Coordinator exposes all missing dependencies at compile time.
[weak self] missing in Combine sink. OrderStatusViewModel subscribes to OrderService.orderUpdates. Capturing self strongly creates a retain cycle. Store the AnyCancellable in a Set<AnyCancellable> and always capture [weak self].

Key Takeaways

The core data flow. Address → restaurant list → menu → basket (server-hosted, locally cached) → order (idempotency key) → SSE stream for live tracking. Say this end-to-end before drawing anything.
SSE is the headline technical decision. Justify it against polling (600 wasted requests) and WebSockets (unidirectional is sufficient). The lifecycle rule: open on screen appear, close on screen disappear.
MVVM eliminates the MVP back-reference. ViewModel has no reference to View. ViewController subscribes to @Published state. Easier to test — no mock View needed.
Coordinator is the DI composition root. No framework needed. Dependencies are composed via manual init at the Coordinator level — missing dependencies are compile errors, not runtime crashes.
Two things to nail in the interview: (1) Justify SSE over polling with numbers — one persistent connection vs 600 requests per order. (2) State architecture rationale before drawing the diagram, then walk the data flow top-to-bottom immediately after.
