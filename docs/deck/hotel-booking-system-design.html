Delta — What This Scenario Adds

Scenario extension of the generic iOS architecture. Read this first — it describes what's the same and what this problem adds on top.

Same as generic architecture

  • Clean Architecture + MVVM + UIKit (plus SwiftUI for some screens — see delta)
  • DTO → Mapper → Domain Model
  • FetchPolicy (.fresh / .cached / .strict) on all Repository reads
  • Typed Param structs on every UseCase
  • @MainActor on ViewModel — all state mutations on main thread, no DispatchQueue.main.async
  • defer { isLoading = false } — guaranteed cleanup on success and failure
  • [weak self] in all closures to avoid retain cycles
  • Coordinator-based navigation; app-scoped services registered at AppCoordinator level
  • Mock-the-layer-below testing strategy
  • async/await for I/O; Combine for reactive binding to @Published state
  • ThirdPartyDataSource (SDK facade) — wraps third-party SDKs as a RemoteDataSource; app calls protocol, never SDK directly
  • Idempotency keys on mutations — client-generated UUID at Param call site for any retryable mutation
  • HTTP 409 ≠ 5xx — concurrency conflicts and transient server errors must never share a code path
  • Infrastructure layer (Gateway suffix) — Domain defines protocol; concrete in Infrastructure; nothing depends on Gateway except DI wiring in Application

Key decisions unique to this scenario

ReservationService must be app-scoped. It owns the live 15-min countdown timer. If owned by a ViewModel, it deallocates when the screen pops — the hold state is lost. Register at app startup in the DI container.
Server owns the clock. expiration_time is always server-generated. The client never calculates a hold window — it only displays a countdown using the server's timestamp.
Offset-limit over cursor pagination. BE controls sort order; hotel search results don't change mid-scroll (no cross-device sync). Offset is simpler and sufficient. Cursor pays off only when data moves (e.g. music library synced across devices).
WebSockets rejected twice. For autocomplete and for the hold timer — HTTP + client countdown achieves the same UX at a fraction of the cost at 10M DAU.
Image cache and offline reservation storage must never share storage. Cache pressure must not evict user reservation data. ImageFileDataSource for images and ReservationLocalDataSource for reservations are completely separate.
local_id is always client-generated. Standard industry idempotency key. Prevents duplicate charges on network retry. Server dedup would add server complexity; client UUID is cheap and reliable.

API Design

Endpoints

MethodRoutePurpose
GET/hotelsSearch + paginated hotel list
GET/hotels/:hotel_idHotel detail
GET/amenitiesFull amenity library (batch fetch, on app launch)
POST/reservationsCreate reservation (places hold)
POST/reservations/paymentFinalize payment

Key Request / Response Shapes

// GET /hotels — query params
destination=<string>
check_in=<string>
check_out=<string>
guest_count=<int>
offset=<int>
limit=<int>          // 25 per page

// POST /reservations — request
CreateReservationRequest {
  local_id: String       // client-generated UUID — idempotency key
  hotel_id: String
  room_id: [String]      // supports multi-room booking
  guest_count: Int
}

// POST /reservations — response
ReservationDTO {
  reservation_id: String
  expiration_time: String    // server-authoritative 15-min timestamp
}

// POST /reservations/payment — request
PaymentRequest {
  payment_token: String    // token from Stripe SDK — not raw card data
}
local_id is always client-generated. UUID created at call site before the request is built. If the request fails and retries, the same local_id is sent — server deduplicates and returns the existing reservation. Prevents double-charges on network retry.
expiration_time is always server-generated. Client never calculates a hold window. Client renders a countdown from the server's timestamp — purely cosmetic. The server lock is authoritative.
Why offset-limit over cursor pagination? Cursor pagination prevents skips/duplicates when data moves mid-scroll (e.g. music library synced across devices). Hotel search results don't move — BE controls sort order and the result set is stable. Offset is simpler and sufficient. The cost of offset (potential for duplicates on live data) does not apply here.

Amenity icons are standardized across the hotel chain. Fetched via GET /amenities on app launch (one batch call). Stored in AmenityLocalDataSource. async/await handles I/O. Combine handles reactivity. Domain Services expose AnyPublisher for state the ViewModel needs to observe long-term (e.g. hold timer countdown). Repositories use async/await internally — no Combine all the way down.

Data Flow

Search + Hotel List

Pattern A — two awaits in the ViewModel. async/await returns once; a single execute() cannot update state twice.

SearchViewModel.search(param:)
    → isLoading = true

    // Phase 1 — cache (instant)
    if let cached = try? await SearchHotelsUseCase.execute(policy: .strict, param:)
        → HotelRepository checks HotelLocalDataSource only — throws on miss
        → ViewModel maps [HotelListing] → UIModel
        → @Published update → view renders immediately from cache

    // Phase 2 — network (background)
    let fresh = try await SearchHotelsUseCase.execute(policy: .fresh, param:)
        → HotelRepository fetches HotelRemoteDataSource.get("/hotels", query:)
        → HotelListingsDTO → Mapper → [HotelListing]
        → HotelLocalDataSource.save(dtos)
        → ViewModel maps [HotelListing] → UIModel
        → @Published update → view refreshes with latest

    → defer: isLoading = false

Hotel Detail

Pattern A — two awaits in the ViewModel.

HotelDetailViewModel.load(hotelId:)
    → isLoading = true

    // Phase 1 — cache (instant)
    if let cached = try? await FetchHotelDetailUseCase.execute(policy: .strict, param:)
        → HotelRepository checks HotelLocalDataSource only — throws on miss
        → ImageService.loadImage(url:) for cached gallery thumbnails
        → ViewModel maps Hotel + resolved images → UIModel
        → @Published update → view renders immediately from cache

    // Phase 2 — network (background)
    let fresh = try await FetchHotelDetailUseCase.execute(policy: .fresh, param:)
        → HotelRepository fetches HotelRemoteDataSource.get("/hotels/:id")
        → HotelDTO → Mapper → Hotel
        → HotelLocalDataSource.save(dto)
        → ImageService.loadImage(url:) for updated gallery
        → ViewModel maps Hotel + resolved images → UIModel
        → @Published update → view refreshes

    → defer: isLoading = false

Create Reservation

ReservationViewModel.reserve(param:)
  → CreateReservationUseCase.execute(param:)   // param.localId = UUID()
      → ReservationRepository
          1. ReservationRemoteDataSource.post("/reservations", body:)
             → ReservationDTO → Mapper → Reservation
          2. ReservationLocalDataSource.save(dto)  // write offline record before navigating
      → returns Reservation
  → ReservationService.startHold(reservation:)   // begins 15-min countdown using server's expiration_time
  → Coordinator pushes PaymentViewController   // reservation.reservationId injected via init

Payment

PaymentViewModel.pay()
  → PaymentService  // Domain Service — app-scoped singleton
      1. paymentGateway.collectToken()   // StripePaymentGateway (Infrastructure) — presents Stripe UI, returns token
      2. → ProcessPaymentUseCase.execute(param: PaymentParam(token:))
             → PaymentRepository (PaymentRepositoryProtocol)
                 → PaymentRemoteDataSource.post("/reservations/payment", body: { payment_token })
      → returns success
  → Coordinator navigates to confirmation screen
Coordinator owns the handoff from Reservation → Payment. reservation_id from the POST response is passed via dependency injection when initializing PaymentViewModel. Neither ViewModel reaches into the other. Alternatively, PaymentViewModel can subscribe to ReservationService using the ID to track hold status — Coordinator owns the wiring either way.

Deep Dives

1. Image Caching — Two-Tier Cache

Problem: Gallery images (thumbnails + full-size) are numerous and frequently accessed. URLSession default cache is not granular enough for TTL eviction or LRU size control.

Cache lookup flow

Image Request (via ImageService)
     │
     ▼
ImageFileDataSource (disk)
     │ hit? → serve immediately (zero network)
     │ miss?
     ▼
MediaRemoteDataSource (S3 fetch)
     │
     ▼
Write binary file to disk (ImageFileDataSource)
Write metadata to CoreData (ImageLocalDataSource):
  { url: String, filePath: String, savedAt: Date }

TTL eviction

Runs on DispatchQueue.global(qos: .background) — triggered on app launch or periodic interval while the app is active. Queries CoreData for savedAt < now - TTL, deletes matching files and records. Never touches the main thread.

Why not BGTaskScheduler? BGTaskScheduler runs when the app is suspended — useful for full background sync. For cache eviction triggered on launch or while the app is active, DispatchQueue.global is sufficient and significantly simpler. Reach for BGTaskScheduler only when the eviction must run without the user opening the app.

LRU size cap

Alongside TTL, a size threshold prevents disk bloat. When the cap is hit, oldest entries are pruned first (LRU-style) regardless of TTL. Thumbnails and full-size images are cached independently — both sizes available for their respective UIs without re-fetching.

CoreData as metadata index, not blob store. CoreData stores file paths and TTL data. Binary image files live on disk managed by ImageFileDataSource. Never shove binary data into a CoreData attribute — it degrades performance and defeats the purpose of file-level LRU eviction.

2. Offline Reservation Storage

Scope: Read-only offline access — user can view existing booking details without network. Creating a reservation always requires a live server lock. No offline creation, no sync flow.

When it's written

Immediately after a successful POST /reservations response, before navigating to Payment:

// Inside ReservationRepository — after mapping to Domain model:
let reservation = Mapper.toDomain(dto)
await ReservationLocalDataSource.save(dto)   // write OfflineReservationDTO to CoreData
return reservation
Why write before navigating to Payment? If the app crashes or is killed during the payment flow, the reservation record is already persisted. User can view the hold details offline and resume payment on re-launch.

3. Search Autocomplete — Local-First

Problem: Real-time character-by-character fetching at 10M DAU is prohibitively expensive server-side. WebSockets were considered and rejected.

User types
     │
     ▼
AmenityLocalDataSource / search cache
(prefetched popular destinations + hotel names on launch)
     │ hit? → show immediately (zero latency)
     │ miss?
     ▼
HotelRemoteDataSource.get("/hotels?destination=<substring>")
(debounced — does not fire on every keystroke)
     │
     ▼
Display results + cache response locally
Prefetch on launch. Popular locations and high-traffic hotel names are fetched once at app start via FetchAmenitiesUseCase (or a dedicated prefetch use case), stored in local DB. Covers the majority of queries for free — zero network, zero latency.
Why not WebSockets? An open persistent connection per active user at 10M DAU is prohibitively expensive server-side. Local prefetch + debounced GET achieves the same UX at a fraction of the infrastructure cost. WebSockets are only justified when the server must push data to the client that the client cannot predict — autocomplete doesn't qualify.

4. Reservation Holds — Server-Owned Lock

Problem: Two users booking the same room simultaneously. 15-minute payment window must be enforced reliably.

Client: CreateReservationUseCase → POST /reservations
     │
     ▼
Server: Lock room_id + hotel_id
Server: Generate expiration_time (now + 15 min)
     │
     ▼
Client: Receive reservation_id + expiration_time
Client: ReservationService.startHold(reservation:)
        → starts local countdown using server's expiration_time
     │
     ├── Payment completed before expiry → POST /reservations/payment ✓
     │
     └── Timer expires / concurrency conflict
              │
              ▼
         HTTP 409 Conflict → "room no longer available" → redirect to Hotel Detail
         HTTP 5xx / timeout → generic network error UX
         (two distinct paths — never conflate them)
Client timer is cosmetic only. The server clock is authoritative. The client countdown tells the user how long they have — it has no bearing on the actual lock on the server. On network failure, the server decides whether the hold is still valid.
HTTP 409 ≠ 5xx. A 409 means a specific concurrency event — another user took the room. This requires a specific UX path ("room no longer available", redirect to Hotel Detail). A 5xx or timeout is a generic network failure — show a retry UI. Conflating them produces a confusing user experience.

5. Payment Processing — Stripe Facade

Why Stripe? App must never handle raw card data. Stripe SDK collects card info, validates, and returns a token. The app passes the token to the backend — never the card number.

User enters card details → Stripe SDK UI
     │
     ▼
Stripe SDK returns payment_token
     │
     ▼
PaymentService → ProcessPaymentUseCase
     → POST /reservations/payment { payment_token }
     │
     ▼
BE charges card server-side via Stripe API
Facade over SDK — always. StripePaymentGateway (Infrastructure) wraps the Stripe SDK behind PaymentGatewayProtocol. PaymentService (Domain Service) orchestrates without importing the SDK. Rest of the app calls PaymentServiceProtocol — never the Stripe SDK directly. Swapping payment providers = one file in Infrastructure (StripePaymentGateway). No ViewModel knows which payment SDK is running.
In Clean Architecture terms: PaymentService (Domain Service) calls PaymentGatewayProtocol.collectToken() (implemented by StripePaymentGateway in Infrastructure — the only class that imports the Stripe SDK), then delegates to ProcessPaymentUseCase → PaymentRepository: PaymentRepositoryProtocol → PaymentRemoteDataSource → API. PaymentViewModel calls PaymentService via PaymentServiceProtocol — nothing above the DataSource level knows which payment SDK is running.

Key Takeaways

Idempotency key — always client-generated for mutation requests that could be retried. UUID at the call site. Same ID on retry = server deduplicates = no double charge.
Server-authoritative timestamps — never trust the client clock for business-critical windows. Hold expiry, pricing windows, time-limited offers — all server-generated, client reads only.
Facade over third-party SDKs. StripePaymentGateway (Infrastructure) wraps Stripe SDK behind PaymentGatewayProtocol; PaymentService (Domain Service) orchestrates without importing the SDK. ImageService wraps caching logic. Rest of the app calls protocols. Swapping providers = one file change.
Prefetch on launch for data expensive to fetch per-keystroke but bounded in size — autocomplete destinations, amenity icon library. Zero-latency lookup, fraction of the network cost.
CoreData as metadata index, not blob store. Store file paths and TTL timestamps. Binary files live on disk. Use CoreData for queryable metadata, not raw binary payloads.
Common gotcha — ReservationService scope. If owned by a ViewModel, it deallocates when the screen pops — the hold countdown stops and the reservation state is lost. Register it as a singleton in your DI container at app startup, not inside a ViewController.
Common gotcha — conflating 409 with 5xx. HTTP 409 Conflict is a specific concurrency event requiring a specific UX (room taken, go back). HTTP 5xx is a generic server error requiring a retry UI. Never conflate them — the user experience is completely different.
Interview verdict: Strong hire for Senior iOS. Candidate communicated trade-offs clearly, avoided over-engineering (rejected WebSockets twice), and kept complexity proportional to requirements.
