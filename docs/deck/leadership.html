How to Lead

Drive the interview. Don't wait to be driven.

Mental Model

You're the Tech Lead. They're the Stakeholder.

They hand you a requirement. You run the design meeting. They listen, probe, and push back — but you own the direction.

Opening — Frame Before Drawing

Don't say
Ok so let me start drawing the layers...
Say this instead
Before I draw anything — let me make sure I understand the problem fully, because the architecture decisions depend heavily on the constraints. Can I ask a few questions first?

You told them you don't jump to solutions. That's the first senior signal.

Narrate Every Decision

Never draw silently. Every box on the board, say why.

I'm putting the UseCase here in the Domain layer — not in the ViewModel — because if I ever swap UIKit for SwiftUI, the business logic doesn't move.
I'm choosing async let here instead of a TaskGroup because there are exactly two fetches. TaskGroup is for N — when you don't know the count at compile time.

They're not grading the diagram. They're grading the reasoning. The diagram is the visual aid.

Handling Push Back

They will push back intentionally — it's a probe, not a correction. Don't fold. Don't get defensive. Acknowledge, then hold or adjust with a reason.

Example — they ask: "Why not just use the Domain model directly in the ViewController?"

Fair question — for a simple CRUD app I'd consider it. The reason I'm adding a UIModel here is that this feed has formatted durations, placeholder image logic, and localized strings. If that lives in the Domain model, it becomes a presentation concern inside business logic. That's the trade-off I'm making. If the team prefers to keep it simpler, I can work with that — I'd just want to be consistent.

Acknowledged their point. Explained the trade-off. Signaled flexibility.

Timebox Yourself Out Loud

Around the 25-minute mark:

I've got the core architecture laid out. I want to make sure we have time to cover concurrency and the cache layer — which I think are the most interesting parts. Should I go deeper here or move on?

Shows you're aware of time. Shows you know what's complex. Gives them control over depth.

Anticipate the Next Question

After every major decision, preempt the follow-up.

...and the obvious question here is what happens when the cache is stale. I handle that with FetchPolicy. Three modes: .fresh always hits the network and saves. .cached returns cache if it exists, falls back to network. .strict returns cache-only and throws on a miss — useful for offline mode where you want an explicit signal rather than a silent empty state.

You answered a question they were about to ask. That's leading.

Bring Up Observability Unprompted

Around minute 50, without being asked:

One thing I always add at the senior level — observability. I'd wire an AnalyticsServiceProtocol into the Domain layer. Events are typed enums with associated values — compiler catches typos and associated values carry structured context. The concrete implementation is in the Application layer — swap ConsoleAnalyticsService for Firebase or Segment without touching business logic. Beyond analytics, crash reporting at the AppDelegate level and performance tracing on the repository layer for slow network calls.

Most candidates wait to be asked. You brought it up.

Close With a Forward Question

Don't stop when the diagram is done:

What's the biggest scalability concern on the current app that this design should address? I want to make sure I haven't optimized for the wrong thing.

Shows you care about the actual problem, not just showing off patterns.

Phrases That Signal Seniority

Let me clarify before I commit to this — it changes the design.
I'm making a trade-off here. The alternative is X, but I prefer Y because...
The interesting engineering problem in this feature is actually...
I want to flag a risk: if the requirement changes to real-time, this part of the design needs to change.
Should I go deeper here or is the concept clear and we move on?
That's a valid concern — here's how I'd handle it...

What Kills Leadership

Avoid these

  • Drawing silently for more than 60 seconds
  • Asking "Is this correct?" — you decide and defend
  • Starting with login or simple screens — go to the hardest piece first
  • Waiting to be asked about caching, concurrency, analytics — bring them up yourself
  • Changing your answer the moment they push back without giving a reason

The One Rule

Mid-level answers the question asked.
Senior anticipates the next question and names the trade-off before the interviewer points it out.

Every decision on the board: "I'm choosing X over Y because..." — before they ask.
