API Design

Key Decisions

  • REST + JSON, authentication assumed
  • Cursor-based pagination over offset — prevents skips/duplicates when library order changes mid-scroll (desktop sync)
  • Sort as API param, not client-side — client doesn't have full library context at 100M DAU scale
  • stream-info is a separate endpoint — keeps manifest URL short-lived without invalidating item metadata
Why REST over GraphQL? REST maps to predictable URLs that CDNs and HTTP caches understand natively. GraphQL POST requests are harder to cache and add client complexity (query language, schema). For a mobile app with well-defined screens and known data shapes, REST is sufficient. GraphQL pays off when many clients need different field subsets — not the case here.
Why JSON over Protobuf? JSON is human-readable (debuggable in Charles/Proxyman), natively supported by Codable, and fast enough for library-sized payloads. Protobuf wins on bandwidth for very high-frequency or very large payloads — not a bottleneck for this design.

Data Model

struct LibraryItem {
    id: UUID
    collectionSummary: CollectionSummary
    updatedAt: Date
    lastListenAt: Date       // drives "sort by recent"
}

struct CollectionSummary {
    id: UUID
    title: String
    itemCount: Int
    type: playlist | album | podcast
    artworkUrl: URL
}

// Polymorphic — enum in Swift
enum PlayableItem {
    case track(Track)      // always starts from beginning
    case episode(Episode)  // resumes from resumeTimestamp
}

struct Track   { id, title, artist }
struct Episode { id, title, artist, resumeTimestamp: TimeInterval }

struct PlayableAsset {    // streaming representation — not for list views
    itemId: UUID
    streamInfo: StreamInfo
}

struct StreamInfo {
    manifestUrl: URL
    expiresAt: Date       // client must refresh before expiry
}

struct PlaybackState {
    status: idle | playing | paused | seeking
    asset: PlayableAsset?
    queue: PlaybackQueue?
}
Why enum over protocol/class inheritance for PlayableItem? Enum gives exhaustive switch — the compiler forces you to handle every case. Protocol polymorphism lets new types slip in silently; class inheritance adds coupling and shared mutable state. For a closed set of types (track, episode) enum is the right tool — new cases are a compile error, not a runtime surprise.
PlayableItem vs PlayableAsset: PlayableItem is the list representation (title, artist, artwork). PlayableAsset is the streaming representation (signed URL, expiry). Keep them separate — list views should never receive short-lived signed URLs they don't need.
Why Track and Episode are separate types: Episode requires resumeTimestamp — it must resume from where the user left off. Track always starts from zero. The enum makes the compiler enforce this distinction at every call site.

Streaming — HLS (Adaptive Bitrate)

Why HLS over progressive download (plain MP3/AAC)? Progressive download streams a single file from start to finish — no quality switching. If the network drops, playback stalls immediately. HLS chunks the audio and lets the player switch quality levels between chunks. At 100M DAU on mobile networks, adaptive bitrate is non-negotiable. Progressive download is fine for short clips or controlled environments; not for a global music app.
Why HLS over DASH (Dynamic Adaptive Streaming over HTTP)? Both are adaptive bitrate protocols. HLS is Apple's standard — AVPlayer has first-class native support with zero extra libraries. DASH requires a third-party player (e.g. ExoPlayer on Android). On iOS, always default to HLS unless there's a specific reason (e.g. cross-platform SDK requirement).
// Client calls /item/{id}/stream-info → gets manifestUrl (.m3u8 file)

Manifest:
  { highQuality: [URL, ...], medium: [URL, ...], ... }

Each quality bucket → chunks:
  .../[quality]/[chunkId]   ← 2–10 second audio segments

Why short-lived URLs (expiresAt)? Direct audio URLs can be extracted from a packet sniffer and shared. Signing them with an expiry means a leaked URL becomes useless quickly. Client refreshes stream-info before expiry during playback.

Adaptive bitrate: Player monitors bandwidth and switches quality levels mid-stream by requesting chunks from a lower/higher quality playlist — no full restart needed.

High-Level Architecture

Pattern: MVVM + Clean Architecture — Presentation → Domain ← Data.

Why MVVM over MVP? In MVP the Presenter holds a reference back to the View via protocol, creating two-way coupling. MVVM breaks that — ViewModel exposes @Published state and doesn't know the View exists. ViewModels are easier to test (no mock View needed) and bind naturally to Combine. MVP made sense before reactive frameworks; MVVM is the current iOS standard.
Why MVVM over VIPER? VIPER splits a screen into 5 objects (View, Interactor, Presenter, Entity, Router). The overhead is justified for very large teams where each layer is owned by different people. For a solo/small team the boilerplate slows you down with no real benefit. MVVM + UseCase gives the same testability with half the files.
Why Clean Architecture over MVC? MVC in iOS degrades to Massive ViewController — business logic, networking, and layout end up in the same file. Clean Architecture enforces a dependency rule (Presentation → Domain ← Data) that makes each layer independently testable and replaceable. The cost is more files; the benefit is every layer can be tested without the others.
Why UIKit over SwiftUI? UIKit gives fine-grained control over scroll performance, custom transitions, and AVPlayerViewController integration — all critical for a music app. SwiftUI's List and animation model can't yet match UIKit for complex interactive layouts. SwiftUI is right for simple new screens; UIKit is preferred when you need full control over performance and native media playback.
Why GRDB over Core Data? Core Data has a complex concurrency model (NSManagedObjectContext) and generates verbose boilerplate. GRDB is plain SQLite with a Swift-first API — you write SQL you understand, get Combine publishers out of the box, no magic behind the scenes. Core Data wins if you need CloudKit sync or have an existing stack. For greenfield, GRDB is simpler and faster to iterate on.

Data Flow — Library Screen

LibraryViewModel.load(policy: .cached)
  → FetchLibraryUseCase.execute(policy:param:)
    → LibraryRepository
        1. localDataSource.fetch()     // return cached rows → UI renders instantly
        2. remoteDataSource.fetch()    // hit API in parallel
        3. write DTOs → LocalDataSource via LibraryItemMapper
        4. LocalDataSource emits stream → ViewModel updates UI
Why FetchPolicy over hardcoding network/cache logic per ViewModel? Without a policy, every ViewModel independently decides whether to hit the network — leading to inconsistent behaviour and duplicated cache logic. FetchPolicy centralises the decision into a typed value that travels ViewModel → UseCase → Repository. The caller declares intent (fresh / stale OK / cache only), the Repository enforces it. Easy to test, easy to change globally.
GRDB as Single Source of Truth: ViewModels never read directly from the network. All reads come from LocalDataSource. Repos write API responses into the DB, the DB emits a stream back up. Instant UI on launch (stale data), silently refreshed when API responds.

PlayerService

Owns PlaybackState, queue, shuffle/repeat logic. App-scoped singleton — must outlive any single screen.

Common gotcha: if PlayerService is owned by PlayerViewModel, it deallocates when the Player screen pops — the music stops. Register it in your DI container at app startup, not inside a ViewController.

StreamRefreshService

Pure logic — given StreamInfo.expiresAt, decides when to trigger FetchStreamInfoUseCase. No network calls itself. Checks if the current time is within a refresh threshold (e.g. 60s before expiry).

Background audio and PlayerService scope are the same problem. Both require the service to outlive any single screen. Solve the scope problem once and both are solved.

Streaming & Playback

HLS Flow (recap)

PlayerService triggers playback
  → PlaybackAssetResolver checks DownloadRepository (DownloadRepositoryProtocol)
      ├─ downloaded? → file:// URL → AVPlayer
      └─ not downloaded? → FetchStreamInfoUseCase → DownloadRepository (DownloadRepositoryProtocol) → MediaRemoteDataSource → HLS CDN
                             → manifest (.m3u8)
                             → quality buckets → chunks (2–10s each)
                             → AVPlayer streams chunks adaptively

manifest expiresAt — Refresh During Playback

  • PlayerEngine checks expiresAt before initiating each new chunk request
  • If near expiry → background task fetches a fresh stream-info via FetchStreamInfoUseCase → MediaRemoteDataSource
  • Refresh happens without interrupting the active audio buffer — chunks already buffered keep playing while the new manifest loads
  • Owner: PlayerService / PlayerEngine, not the ViewModel

Offline Playback — Two Paths

ScenarioApproach
Standard downloaded filesPlaybackAssetResolver returns file:// URL directly to AVPlayer — no extra work
Protected / chunked formatCustom AVAssetResourceLoadingDelegate intercepts requests and serves local binary data

DownloadLocalDataSource holds the itemId → localFilePath mapping; PlaybackAssetResolver (inside PlayerService) accesses it via DownloadRepository.

Why file:// URL over AVAssetResourceLoadingDelegate? AVAssetResourceLoadingDelegate lets you intercept every network request AVPlayer makes and serve data yourself — useful for DRM, encryption, or proprietary chunk formats. But it adds significant complexity (manage loading requests, handle cancellation, byte-range requests). For standard downloaded audio files a plain file:// URL is sufficient and far simpler. Use the delegate only when the file format or protection requires it.

Background Audio

  • App must declare audio background mode in project settings
  • AVAudioSession category set to .playback — allows audio to continue when app backgrounds
  • PlayerService must be app-level singleton (not owned by a ViewModel) so the audio thread stays alive across screen transitions and backgrounding
Background audio and PlayerService scope are the same problem. Both require the service to outlive any single screen. Solve the scope problem once and both are solved.

Technical Deep Dives

1. Pagination — offset vs cursor

Offset-based (simple, breaks on live data)

GET /me/library?page=2&limit=20
→ server: SELECT * FROM library OFFSET 20 LIMIT 20

Page 1:  [A, B, C, D, E, F, G, H, I, J]
                       ↑
           New item "X" inserted here (desktop sync)

Page 2:  [F, G, H, I, J, K, ...]  ← F already seen — duplicate

Cursor-based (stable across insertions)

GET /me/library?cursor=eyJpZCI6IkYifQ&limit=20
→ server: SELECT * FROM library WHERE id > :cursor LIMIT 20

Page 1:  [A, B, C, D, E, F, G, H, I, J]  → nextCursor points to J
                       ↑
           New item "X" inserted here

Page 2 using cursor→J:  [K, L, M, ...]  ← starts after J, unaffected

The cursor is an opaque string — the client never parses it, just stores and sends it back. The server decides what's inside (ID, timestamp, composite key). This decouples client from server internals.

Why opaque string over a typed cursor struct? Exposing cursor internals (e.g. { id: "abc", timestamp: 123 }) couples the client to server implementation. If the server changes its cursor strategy, every client breaks. An opaque base64 string hides that detail — the client echoes it back without understanding it. Same principle as treating an HTTP ETag as opaque.
CursorOffset
Live-synced data✅ stable❌ skips / duplicates
Static dataOverkill✅ simpler
Jump to arbitrary page❌✅
Server implementationHarderEasy
Rule: cursor when data moves (library synced across devices), offset when it doesn't (search results, static lists).

2. What is HLS and why does it exist?

Why HLS over progressive download (plain MP3/AAC)? Progressive download streams a single file from start to finish — no quality switching. If the network drops, playback stalls immediately. HLS chunks the audio and lets the player switch quality levels between chunks. At 100M DAU on mobile networks, adaptive bitrate is non-negotiable.
Why HLS over DASH? Both are adaptive bitrate protocols. HLS is Apple's standard — AVPlayer has first-class native support with zero extra libraries. DASH requires a third-party player on iOS. Always default to HLS unless there's a specific cross-platform SDK requirement.

HLS (HTTP Live Streaming) solves one problem: audio quality degrades or buffers on slow networks.

The naive approach — serve one large audio file — falls apart because you can't switch quality mid-way, the whole file must download before playback starts, and a leaked URL gives permanent access.

HLS solves all three by splitting audio into small chunks and letting the player pick quality dynamically.

3. How HLS works — step by step

1. Client calls GET /item/{id}/stream-info
   → receives manifestUrl (.m3u8 file)

2. Client fetches manifest from CDN:
   #EXT-X-STREAM-INF:BANDWIDTH=128000
   https://cdn.example.com/low/index.m3u8

   #EXT-X-STREAM-INF:BANDWIDTH=320000
   https://cdn.example.com/high/index.m3u8

3. Client picks a quality, fetches its chunk playlist:
   #EXTINF:6.0,
   https://cdn.example.com/high/chunk_001.aac
   #EXTINF:6.0,
   https://cdn.example.com/high/chunk_002.aac
   ...

4. Client downloads chunks one at a time, plays in sequence
5. Bandwidth drops → switch to low/ playlist → next chunk is lower bitrate
   No restart. Seamless.
Key insight: the client never downloads the whole file. It only ever has a few seconds of buffer ahead of the playhead.

4. AVPlayer and HLS on iOS

You write almost no streaming code yourself — AVPlayer handles it all.

// This is all you need for HLS streaming
let url = URL(string: "https://cdn.example.com/manifest.m3u8")!
let player = AVPlayer(url: url)
player.play()

AVPlayer internally: fetches and parses the manifest, monitors bandwidth, switches quality levels automatically, buffers chunks ahead of playback, retries failed chunk downloads.

Your job: give it the right URL and configure AVAudioSession.

5. AVAudioSession — required for real playback

Without this, audio stops when the screen locks or the app backgrounds.

6. File storage — three tiers

Tier 1 — Memory buffer
AVPlayer owns this. A few seconds of decoded audio ahead of the playhead. Freed on track change. You never touch it.
Tier 2 — LRU disk cache
Recently streamed chunk files. Max 5 GB, evicts least-recently-used when full. If user replays a track, skip the network. You manage this via DownloadService.
Tier 3 — Explicit download
User tapped "Save for offline." Full track stored permanently. Never evicted — only removed when user deletes. Separate from Tier 2.
Why Tier 2 and Tier 3 must be separate stores: if they shared storage, streaming cache pressure could evict a track the user explicitly saved for offline. Separate LocalFileDataSource instances prevent this entirely.
Why LRU over FIFO or LFU eviction? FIFO evicts the oldest item regardless of recent use — you'd evict an album played daily. LFU favours popular items but punishes newly added content and is expensive to track. LRU evicts what hasn't been touched recently — a good proxy for "user probably doesn't need this." Simple to implement, good real-world performance for media caches.
Why 5 GB? Roughly 40–60 full albums at 128kbps. Enough for meaningful background caching without dominating a typical iPhone's storage. Expose it as a user setting ("Storage limit") so users can tune it.

7. Offline playback — how AVPlayer reads local files

When a track is fully downloaded, skip the CDN entirely and hand a file:// URL to AVPlayer:

// Online path
let url = URL(string: "https://cdn.example.com/manifest.m3u8")!

// Offline path — AVPlayer doesn't care which one it is
let url = URL(fileURLWithPath: "/var/mobile/.../track_123.m4a")

AVPlayer(url: url).play()

PlaybackAssetResolver (inside PlayerService) makes the decision via DownloadRepository (injected as DownloadRepositoryProtocol):

DownloadRepository.localFilePath(for: itemId)  // protocol call — Domain knows nothing about DownloadLocalDataSource
  ├─ found  → file:// URL  (offline)
  └─ nil    → FetchStreamInfoUseCase → DownloadRepository → MediaRemoteDataSource → HLS manifest URL  (streaming)
Why file:// URL over AVAssetResourceLoadingDelegate? AVAssetResourceLoadingDelegate lets you intercept every request AVPlayer makes and serve data yourself — useful for DRM, encryption, or proprietary chunk formats. But it adds significant complexity (manage loading requests, handle cancellation, byte-range requests). For standard downloaded audio files a plain file:// URL is sufficient and far simpler. Use the delegate only when the file format or protection requires it.
Advanced: if files are stored in a protected/chunked format, use a custom AVAssetResourceLoadingDelegate to intercept requests and serve local binary data. Default to the simple file:// path first.

8. Short-lived URLs — why and how

Why URLs expire

A permanent CDN URL can be extracted from a packet sniffer and shared publicly. A signed URL with an expiry is useless after it expires.

// Server signs the URL
https://cdn.example.com/track_123.m3u8?token=abc&expires=1716490800

// CDN validates token + expiry on every request
// After expiresAt → CDN returns 403

How the client handles it

StreamRefreshService.needsRefresh(for: streamInfo)
  → checks: Date() > streamInfo.expiresAt - refreshThreshold  (e.g. 60s)
  → if yes → background fetch new stream-info from API
             swap manifest URL in PlayerEngine
             buffered chunks already playing are unaffected
The refresh is invisible to the user — chunks already in the buffer play out while the new manifest loads in the background.

9. LRU Cache — how it works

LRU = Least Recently Used. When the cache is full, evict the item accessed least recently.

Cache (max 3 items for illustration):
  [chunk_A (oldest), chunk_B, chunk_C (newest)]

New chunk_D needs caching → cache full → evict chunk_A:
  [chunk_B, chunk_C, chunk_D]

User replays chunk_B → promoted to newest:
  [chunk_C, chunk_D, chunk_B]

In practice (max 5 GB): each chunk is ~50–200 KB. 5 GB holds tens of thousands of chunks — roughly 40–60 full albums. Eviction is by byte size, not item count.

Implement with NSCache (auto-evicts under memory pressure) or a manual SQLite-backed LRU using GRDB.

10. Bad network — adaptive bitrate in detail

AVPlayer measures observed bandwidth and buffer level continuously:

Buffer > 30s AND bandwidth stable  → upgrade quality
Buffer < 10s OR bandwidth dropping → downgrade quality
Buffer = 0s                        → stall

You don't control this — AVPlayer owns it. What you do own:

// Observe stalls — show buffering indicator
player.observe(\.timeControlStatus) { player, _ in
    switch player.timeControlStatus {
    case .playing:               // hide spinner
    case .waitingToPlayAtSpecifiedRate: // show spinner
    case .paused:                // user paused or ended
    }
}

// Data saver mode — cap quality from client side
player.currentItem?.preferredPeakBitRate = 128_000  // cap at 128kbps
player.currentItem?.preferredPeakBitRate = 0         // 0 = no cap (default)

11. Complete network failure — retry strategy

When network cuts out entirely (airplane mode, tunnel):

  • AVPlayer plays out existing buffer (~15–30s)
  • Buffer hits zero → stall → timeControlStatus = .waitingToPlayAtSpecifiedRate
  • AVPlayer automatically retries when network returns — no code needed
  • Playback resumes from where it stalled

What you DO need to handle — distinguish stall from error:

// Track finished normally
NotificationCenter.default.addObserver(
    forName: .AVPlayerItemDidPlayToEndTime, ...
) { _ in // advance queue }

// Actual error — corrupted chunk, 403, bad URL
NotificationCenter.default.addObserver(
    forName: .AVPlayerItemFailedToPlayToEndTime, ...
) { _ in // show error, offer retry }

AVPlayerItemPlaybackStalledBuffer ran dryYes, autoShow spinner only

13. Seeking during streaming

Seeking in HLS is different from seeking in a local file — you're jumping between chunks, not byte offsets.

User drags seek bar to 2:30 (150s)

AVPlayer calculates which chunk contains t=150s:
  chunk_025 = 144–150s  ← target

AVPlayer:
  1. Discards buffered chunks ahead of current position
  2. Fetches chunk_025 from CDN
  3. Resumes buffering forward from chunk_026
  4. Playback resumes at t=150s

The chunk containing the seek target must download before playback resumes — this is the brief pause on slow networks.

Implementing the seek bar

// Drive seek bar position — fires every 0.5s
let interval = CMTime(seconds: 0.5, preferredTimescale: CMTimeScale(NSEC_PER_SEC))
player.addPeriodicTimeObserver(forInterval: interval, queue: .main) { time in
    let current = CMTimeGetSeconds(time)
    let duration = CMTimeGetSeconds(player.currentItem?.duration ?? .zero)
    progress = current / duration   // 0.0 → 1.0
}

// User finishes dragging
func seek(to fraction: Double) {
    guard let duration = player.currentItem?.duration else { return }
    let target = CMTime(seconds: CMTimeGetSeconds(duration) * fraction,
                         preferredTimescale: 600)
    // .positiveInfinity = chunk-boundary accuracy, faster — right for music
    player.seek(to: target, toleranceBefore: .positiveInfinity, toleranceAfter: .positiveInfinity)
}
ToleranceAccuracySpeedUse for
.zero / .zeroFrame-accurateSlowerVideo scrubbing
.positiveInfinityChunk-boundaryFasterMusic ✅
PlaybackState during seek is transient. .seeking must always resolve back to .playing or .paused in the completion handler. Never leave the state machine stuck at .seeking.

Key Takeaways

State principles first, then draw — architecture rationale before the diagram.
Walk the data flow end-to-end after drawing — out loud, top to bottom. Catches missing repo dependencies and edge cases live.
Always wrap the DB — LocalDataSource is this abstraction. Repositories never touch UserDefaults/GRDB directly. ✅
Go deep on complex topics proactively — don't wait to be asked. Signed URL expiry, offline edge cases, stall vs error distinction.
The two things most people get wrong: (1) PlayerService scoped to a ViewController instead of app-level — music stops when screen pops. (2) Conflating a network stall with a playback error — they require completely different handling.
