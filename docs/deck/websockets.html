WebSockets Explained Simply

Real-time communication for mobile — transports compared, WebSocket mechanics, iOS implementation.

Real-Time Transport Options

Push Notifications
Unreliable

APNs/FCM delivery only. No ordering, no guarantee. Use for app wakeup — never as a primary data stream.

HTTP Polling
Wasteful

Client polls on a fixed interval. Most responses are empty. Latency = poll interval. Burns battery and server capacity.

Long-Polling
Expensive

Server holds the request open until data is available. Reduces empty responses, but ties up one server thread per client.

Server-Sent Events (SSE)
Unidirectional

Persistent HTTP stream, server → client only. Great for feeds and tickers. Client must use a separate HTTP request to send data.

WebSockets
Recommended

Full-duplex, persistent TCP channel. Both sides send frames at any time. Use for chat, games, live trading, collaborative editing.

Core Concepts

WebSockets

Established via an HTTP/1.1 upgrade handshake, then the TCP connection is no longer HTTP — it becomes a raw WebSocket channel. Low per-frame overhead (~2–10 byte header vs. full HTTP headers). Either side can send at any time.

WebSocket Frames

The smallest unit of the protocol. Each frame carries an opcode (text, binary, ping, pong, close), a length field, an optional masking key (required client→server), and the payload. A single logical message can span multiple frames, but in practice most messages fit in one. Frames carry either UTF-8 text or raw binary — the receiver interprets bytes according to the opcode.

How It Works

Handshake — HTTP → WS Upgrade

Client → Server: GET /chat HTTP/1.1 Upgrade: websocket Connection: Upgrade Sec-WebSocket-Key: <base64-random> Server → Client: HTTP/1.1 101 Switching Protocols Upgrade: websocket Connection: Upgrade Sec-WebSocket-Accept: <derived-key>

After the 101, the TCP connection is a raw WebSocket channel. Both peers send frames independently.

Frame Lifecycle

  1. Caller invokes send(data).
  2. Data is wrapped in a WebSocket frame (opcode + length + mask + payload).
  3. Receiver strips framing, delivers payload to the application.
  4. Either side sends a CLOSE frame for clean shutdown.

Connection Lifecycle on Mobile

App foreground → connect() → handshake → open ↓ send / receive frames ↓ App backgrounds → close() (explicit teardown) ↓ App foregrounds → reconnect() → handshake → open

Key Patterns

Typed JSON Message Envelope

Wrap every message so the receiver can route it without inspecting raw bytes.

JSON
{
  "type": "chat.message",
  "payload": {
    "roomId": "abc123",
    "text": "hello"
  }
}
Why: Decouples routing from content. New event type = new type string + new handler. No protocol change.

Message Acknowledgement + Sequence Numbers

Assign a monotonically increasing sequence number to each sent message. Receiver sends back an ack. Sender buffers unacked messages and retransmits on timeout.

JSON
{ "type": "ack", "payload": { "seq": 42 } }
Why: TCP guarantees delivery within a session — not across reconnects. App-level acks enable at-least-once delivery after a drop.

Heartbeat (Ping / Pong)

Client sends a WebSocket ping frame every 30 seconds. Server replies with pong. No pong within timeout → treat as dead → reconnect.

Why: NAT gateways and proxies silently kill idle TCP connections (typically 60–300s). Pings keep the path alive and surface dead sockets before you try to use them.

Reconnect with Exponential Backoff

On disconnect: min(base × 2^attempt, maxDelay) + jitter (delay × random(0.5, 1.0)).

Why: Jitter prevents a thundering herd when a server restarts and thousands of clients reconnect simultaneously.

Trade-offs & Design Decisions

Decision Why This Approach What You Give Up
WebSockets over SSE Need client→server messages (chat input, game actions) SSE is simpler; HTTP/2 multiplexes it natively
WSS (TLS) always Prevents MITM; required by many app store policies Slight handshake overhead
Close on background Conserves battery; avoids unpredictable OS teardown Re-establish on foreground adds latency
App-level acks over TCP-only Survives connection drops; enables at-least-once delivery More code; requires message buffer + retry logic
Typed envelope JSON Human-readable, easy to evolve More bytes than binary (Protobuf, MessagePack)
URLSessionWebSocketTask No third-party dependency; full control over reconnect logic Must implement reconnect, queuing, background handling yourself
Starscream Auto-reconnect, message queuing, background logic included External dependency; less control over internals

Code Examples

iOS — URLSessionWebSocketTask

Swift
// Establish a WSS connection using URLSession
let url = URL(string: "wss://api.example.com/chat")!
let task = URLSession.shared.webSocketTask(with: url)
task.resume()

// Send a message
let message = URLSessionWebSocketTask.Message.string(#"{"type":"ping","payload":{}}"#)
task.send(message) { error in
    if let error { print("Send error: \(error)") }
}

// Receive — must be re-armed after every frame
func receiveNext() {
    task.receive { [weak self] result in
        switch result {
        case .success(let message):
            switch message {
            case .string(let text): self?.handle(text)
            case .data(let data):   self?.handle(data)
            @unknown default: break
            }
            self?.receiveNext() // re-arm
        case .failure(let error):
            print("Receive error: \(error)")
            // trigger reconnect logic here
        }
    }
}

iOS — Heartbeat Loop

Swift
// Ping every 30s; reconnect if pong doesn't arrive
func startHeartbeat() {
    Timer.scheduledTimer(withTimeInterval: 30, repeats: true) { [weak self] _ in
        self?.task.sendPing { error in
            if let error {
                print("Heartbeat failed: \(error) — reconnecting")
                self?.reconnect()
            }
        }
    }
}

Common Pitfalls / Gotchas

Forgetting to re-arm receive() — URLSessionWebSocketTask.receive reads exactly one frame. Call it again inside the completion handler or you miss everything after the first message.
No auto-reconnect in URLSessionWebSocketTask — Unlike Starscream, the native API drops and stays disconnected. Detect failure in the receive completion and call reconnect() manually.
Not closing on background — iOS may silently kill a suspended socket. Close explicitly so the server gets the teardown handshake; otherwise it won't know you're gone until heartbeat timeout.
Network interruptions & proxy closure — Most disconnects are Wi-Fi loss. Intermediate proxies also silently drop idle connections. Heartbeats handle the proxy case; reconnect logic handles network loss.
Reconnecting without backoff — A tight retry loop hammers a recovering server. Always use exponential backoff + jitter.
Assuming TCP delivery = app delivery — TCP guarantees delivery within a session, not across reconnects. In-flight frames on a dropped connection are lost. App-level acks are required for cross-session reliability.
WSS cert issues on real devices — URLSession rejects self-signed certs. Test with a valid certificate on device early.
Mixing binary and text frames — Agree upfront on frame type. Unexpected switches cause silent decode failures.

Interview Talking Points

  • Handshake: WebSockets start with HTTP/1.1 101 Switching Protocols, then become a raw TCP channel — this is why they work on ports 80 and 443.
  • Full-duplex: Both sides send frames at any time without waiting — unlike HTTP request/response.
  • Mobile lifecycle: Close on background, reopen on foreground. The OS can silently kill idle sockets.
  • Heartbeats: Ping/pong every ~30s keeps NAT paths alive and surfaces dead connections before you try to use them.
  • Reliability: TCP alone doesn't cover cross-session delivery. App-level acks + sequence numbers are needed for at-least-once guarantees.
  • WSS always: Plaintext WS is blocked by many networks and rejected in app review.
  • iOS choice: URLSessionWebSocketTask = full control, no dependency, but you build reconnect + queuing. Starscream = those batteries included, at the cost of a dependency.
  • Protocol design: JSON envelope with a type field multiplexes multiple event types over one connection without a protocol change.

Key Takeaways

WebSockets are the right tool when you need bidirectional, low-latency communication — they start as HTTP, upgrade to a persistent TCP channel, and let both sides send frames at any time. On mobile, the core discipline is lifecycle management: close on background, reconnect on foreground with exponential backoff, and use heartbeats to catch connections that went silent without a clean teardown. For reliability across drops, TCP alone isn't enough — you need application-level acks and sequence numbers. When choosing between URLSessionWebSocketTask and Starscream, the trade-off is control vs. built-in infrastructure: native gives you less dependency overhead, Starscream gives you auto-reconnect and message queuing out of the box.
