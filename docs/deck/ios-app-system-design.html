Architecture Pattern

Clean Architecture + MVVM + UIKit

Presentation
ViewController + ViewModel (Combine @Published)
Domain
UseCase + Protocol + Model + Param + FetchPolicy
Data
Repository + DataSource + DTO + Mapper + APIClient
Application
AppDelegate + Coordinator + DI (manual init injection)
Layer dependency rule: Presentation → Domain ← Data. Domain depends on nothing. Data knows Domain models (via Mapper). Presentation knows Domain models (via UseCase). Neither knows the other exists.

Why These Choices

Clean Architecture over MVC — MVC in iOS degrades to Massive ViewController: business logic, networking, and layout all end up in one file. Clean Architecture enforces the dependency rule so each layer is independently testable and replaceable. Cost: more files. Benefit: every layer can be tested without the others.
MVVM over MVP — In MVP the Presenter holds a reference back to the View via protocol — two-way coupling. MVVM breaks that: ViewModel exposes @Published state and has no reference to the View. ViewModels are easier to test (no mock View needed) and bind naturally with Combine.
MVVM over VIPER — VIPER splits a screen into 5 objects. The overhead is justified for very large teams where each layer is owned by a different person. For a small team, MVVM + UseCase gives the same testability with half the files.
New app → SwiftUI. SwiftUI is the current default for greenfield iOS development. State-driven rendering and declarative layout fit cleanly with MVVM — the ViewModel exposes @Published state; the View subscribes with no extra Combine boilerplate.
Legacy app → stay on UIKit. Migrating an existing UIKit codebase is high risk for low gain. New screens in a UIKit app should remain UIKit unless there is a specific team mandate to migrate.
UIKit is still the right call even in a new app for: granular scroll lifecycle (willDisplay cell for paginated fetches), complex custom transitions, direct AVPlayerViewController / AVPlayerLayer integration, or pixel-level layout fidelity SwiftUI can't match.
Hybrid UIKit/SwiftUI is valid screen-by-screen. Coordinator wires them together. UIKit hosts SwiftUI via UIHostingController; SwiftUI embeds UIKit via UIViewRepresentable. Rule of thumb: lifecycle control needed → UIKit. Reactive state → UI → SwiftUI.

Layer Breakdown

Presentation

ViewController
Owns ViewModel (strong ref). Binds to @Published state via Combine sink. Calls ViewModel methods on user actions. Never calls UseCases or Repositories directly.
ViewModel
@MainActor — all state on main thread. @Published properties (tracks, isLoading, errorMessage). Calls UseCases. Maps Domain → UIModel. [weak self] in all closures. defer { isLoading = false } on every exit path.
UIModel
Flat, display-ready struct. Never pass Domain models to the View directly — the ViewModel maps Domain → UIModel so the View has zero knowledge of business types.
defer { isLoading = false } — guaranteed cleanup on success and failure. Place it immediately after setting isLoading = true. Never rely on individual return paths to reset state.

Domain

UseCase
One public method: execute(policy:param:) async throws → Output. Stateless — created per call or injected. Orchestrates one business action. Calls Repository methods. Never touches networking or storage directly.
Domain Service
Stateful or long-lived logic not tied to a single user action. Injected, lives as long as needed (often app-scoped). Examples: PlayerService, SessionService, AuthService.
Model
Pure Swift structs — no import UIKit, no Foundation networking types. The only type that crosses all layers.
Param
Typed struct for every UseCase input. Split into path: and query: sub-structs. Adding a field never breaks existing call sites.
FetchPolicy
.fresh always hits network. .cached returns cache if available, else network. .strict cache only, throws on miss. Travels ViewModel → UseCase → Repository. Repository is the only place that interprets it.
UseCaseDomain Service
Triggered byUser actionAnother component
StateStatelessCan be stateful
LifetimePer callInjected, lives as needed
ExampleSearchTracksUseCasePlayerService, SessionService
Why FetchPolicy over hardcoding cache logic per ViewModel? Without a policy, every ViewModel independently decides whether to hit the network — leading to inconsistent behaviour and duplicated cache logic. FetchPolicy centralises the decision into a typed value. The caller declares intent, the Repository enforces it. Easy to test, easy to change globally.

Data

Repository
Implements RepositoryProtocol (Domain interface). Coordinates RemoteDataSource and LocalDataSource. Applies FetchPolicy. Converts DTO → Domain Model via Mapper. Never exposed directly to Presentation.
RemoteDataSource
Wraps APIClient. Builds request URLs and decodes responses. Returns DTOs — never Domain models.
LocalDataSource
Wraps persistence backend (UserDefaults / SQLite / GRDB). Returns DTOs — never Domain models. Repository never touches the storage backend directly.
DTO
Mirrors the API or DB schema exactly. Codable. Disposable — only lives between the network/DB and the Mapper.
Mapper
The only type that knows both DTO and Domain model. static func toDomain(_ dto: DTO) -> Model? Returns Optional — invalid data is silently dropped, never crashes.
ThirdPartyDataSource
SDK facade — wraps a third-party SDK (Stripe, Firebase, etc.) as a RemoteDataSource. Rest of the app calls a RepositoryProtocol — never the SDK directly. Swapping the SDK = one change in one DataSource file. Examples: PaymentRemoteDataSource (Stripe), AnalyticsDataSource (Firebase).

DTO → Mapper → Domain Model flow

RemoteDataSource fetches JSON
  → decoded as DTO               // Codable, mirrors API shape
  → Mapper.toDomain(dto)         // Domain Model, or nil if invalid
  → Repository returns [Model]   // to UseCase
  → UseCase returns [Model]      // to ViewModel
  → ViewModel maps → UIModel    // for the View
Mapper is the seam between external data and your business logic. Keep it the only crossing point. If you find yourself converting DTOs anywhere else, move it to the Mapper.

Application

AppDelegate
Entry point. Wires window and root Coordinator. Registers app-scoped services (analytics, player, auth).
Coordinator
Owns navigation logic — ViewControllers never push/present directly. Creates UseCases and ViewModels (DI composition root). One Coordinator per flow.
DI (manual init injection)
No DI framework. Dependencies passed through init. No default concrete arguments on Repository or UseCase inits. DataSources and APIClient composed at Coordinator level.

Dependency Injection

Dependencies flow inward. Each layer receives its dependencies via init. No layer constructs its own dependencies.

// Coordinator — composition root, builds the full graph
let client           = APIClient()
let remoteDataSource = TrackRemoteDataSource(client: client)
let localDataSource  = TrackLocalDataSource()
let repository       = TrackRepository(remote: remoteDataSource, local: localDataSource)
let useCase          = SearchTracksUseCase(repository: repository)
let viewModel        = TrackListViewModel(searchTracks: useCase)
No default concrete args in Repository inits. A default argument that instantiates a concrete type (= TrackLocalDataSource()) hides the dependency and breaks testability. The default belongs in the DI composition root, not the init signature.
Protocol at every boundary. TrackRemoteDataSourceProtocol, TrackLocalDataSourceProtocol, TrackRepositoryProtocol — replace with mock in tests, real impl in production. No mocking of concrete types.

Generic Data Flows

Read flow — loading a screen

ViewController.viewDidLoad()
  → ViewModel.load()
      → isLoading = true
      → UseCase.execute(policy: .cached, param:)
          → Repository.fetch(policy: .cached, param:)
              → LocalDataSource.fetch()  // cached DTO → fast return
              → RemoteDataSource.fetch() // background network call
              → LocalDataSource.save(dto) // write fresh data to cache
          → returns [Model]
      → maps Model → UIModel
      → @Published state updated → View re-renders
      → defer: isLoading = false

Mutation flow — create / update

ViewController sends user input
  → ViewModel.submit(input)
      → isLoading = true
      → UseCase.execute(param:)
          → validates param  // throws if invalid — before any network call
          → Repository.create/update(param:)
              → RemoteDataSource.post/put() → DTO → Mapper → Model
          → returns Model
      → maps Model → UIModel → updates @Published state
      → defer: isLoading = false
Validate in the UseCase, before the Repository call. The Repository assumes its input is valid — it shouldn't know what "valid" means for the business. The UseCase is the gatekeeper.
Idempotency keys on mutations. For any mutation that could be retried (network timeout, app restart mid-request), generate a client-side UUID before building the request and include it as a field (e.g. localId). If the request is retried with the same UUID, the server returns the existing record rather than creating a duplicate. Applies to reservations, orders, payments — any operation where a duplicate would cause business harm. Generate the UUID at the Param struct call site, not inside the Repository or DataSource.

Networking

APIClient
Generic URLSession-based HTTP client. get<T: Decodable>(_ url:), post<T, B: Encodable>(_ url:, body:), etc. No domain knowledge.
Request structs
One per endpoint: TrackSearchRequest, CreatePlaylistRequest. Carries the fields needed to build the URL and body. Lives at Data/Network/Requests/.
APIError
Typed enum: .invalidURL, .notFound, .networkError, .decodingError, .conflict. Propagates async throws up to ViewModel. ViewModel catches and sets errorMessage: String?.

HTTP status code semantics — map at RemoteDataSource level, never in the ViewModel

StatusMeaningClient action
400Bad Request — malformed requestLog and show generic error
401Unauthorized — session expiredTrigger re-auth flow
403Forbidden — user lacks permissionShow access denied
404Not Found — resource goneRemove from local cache
409Conflict — concurrency event (item claimed by another user)Domain-specific UX — NOT a generic retry
5xxServer error — transientShow retry UI
Network timeoutTransientShow retry UI
409 and 5xx must never share a code path. A 409 means a specific domain event occurred (another user acted first) — it requires domain-specific UX ("room no longer available", "item already claimed"). A 5xx means "try again later." Conflating them produces a confusing user experience and masks real concurrency bugs.
APIClient lives inside Data/Network/ — not a separate layer. It's an implementation detail of RemoteDataSources. Nothing in Domain or Presentation ever touches it.

Persistence

LocalDataSource
Wraps the storage backend. Repositories never touch the backend directly — always through LocalDataSource. Stores and retrieves DTOs, not Domain models. Keyed by request parameters for cache lookup.
BackendUse when
UserDefaultsSimple key-value cache, small payloads, no relational queries needed
GRDB (SQLite)Relational queries, Combine publishers, no magic — preferred for greenfield
Core DataOnly if CloudKit sync required or an existing stack already exists
Why wrap storage in LocalDataSource? The Repository doesn't know or care what's underneath. Swapping UserDefaults for GRDB touches one file — LocalDataSource — and nothing else. The Repository, UseCase, and ViewModel are completely unaffected.

Navigation

AppCoordinator
Root coordinator. Owns tab bar. Handles deep links. Routes to feature coordinators.
FeatureCoordinator
One per flow (e.g. SearchCoordinator, HomeCoordinator). Creates ViewModels and ViewControllers. ViewControllers never call push/present directly.
// Deep link flow
NotificationCenter.default.post(.handleDeepLink, object: link)
  → AppCoordinator.handle(link)
      → selects correct tab
      → delegates to feature coordinator
      → coordinator creates ViewModel + ViewController and pushes
App-scoped services (e.g. PlayerService) must be registered at AppCoordinator level — not owned by any ViewController. If owned by a ViewController, they deallocate when that screen is popped.

Concurrency

@MainActor on ViewModel
All state mutations on main thread. No DispatchQueue.main.async needed.
async/await throughout
No completion handlers. Clean call stack, propagates throws naturally.
async let
Two concurrent fetches with fixed known count.
withThrowingTaskGroup
N concurrent fetches with dynamic count (e.g. fetch detail for each item in a list).
[weak self]
All closures that capture self. Prevents retain cycles between ViewController and ViewModel.
defer { isLoading = false }
Cleanup on any exit path — success, throw, or early return. Place immediately after isLoading = true.
async letwithThrowingTaskGroup
Use whenFixed small N (2–3 concurrent)Dynamic N
SyntaxCleaner, inlineMore explicit, loop-based
ExampleFetch tracks + playlists in parallelFetch detail for each item in a list

Testing Strategy

Rule: mock the layer below, assert on the layer you just built.
LayerWhat to mockWhat to assert
ViewModelMockUseCase@Published state after user action
UseCaseMockRepositoryReturn value, thrown error, param forwarding
RepositoryMockRemoteDataSource + MockLocalDataSourceFetchPolicy logic, Mapper output, cache read/write order
DataSourceMockAPIClient / in-memory storeRequest shape, URL construction, response decoding
// Mock pattern — same shape for every layer
final class MockSearchTracksUseCase: SearchTracksUseCaseProtocol {
    var stubbedResult: Result<[Track], Error> = .success([])
    private(set) var lastParam: SearchTracksParam?

    func execute(policy: FetchPolicy, param: SearchTracksParam) async throws -> [Track] {
        lastParam = param
        return try stubbedResult.get()
    }
}
No mocking of concrete types. Every dependency is injected via a protocol — replace with mock in tests, real impl in production. If you can't swap it, the DI is wrong.

Adapting to a Scenario

When given an interview problem, map it onto this architecture in order:

1. Identify the domain
What are the entities? (Track, Playlist, User, Order, Ride…)
2. Name the Repositories
One per domain entity or aggregate. Each gets a Remote + Local DataSource.
3. Name the UseCases
One per user action or screen load. Stateless, single execute method.
4. Identify Domain Services
Anything stateful, long-lived, or shared across screens. App-scoped.
5. Apply FetchPolicy
Does this screen need fresh data? Can it show stale? Offline-first?
6. Identify local storage need
Cache only? Offline-first with user-controlled saves? File storage? LRU eviction?
7. Draw the data flow
Top to bottom, out loud, immediately after the diagram. Catches missing dependencies and edge cases live.
State your principles first, then draw. Before touching the whiteboard: "I'll use Clean Architecture — Presentation → Domain ← Data. Domain depends on nothing. Let me name the layers for this scenario." One sentence. Then draw.
